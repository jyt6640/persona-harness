# Architecture Evolution

이 문서는 프로젝트 구조를 어떻게 진화시킬지에 대한 기준을 정의한다.

아키텍처는 처음부터 완벽하게 설계하는 대상이 아니라,
현재 문제를 해결하며 점진적으로 발전시키는 구조로 본다.

---

## 핵심 방향

- 현재 문제를 우선 해결한다.
- 구조는 실제 문제 신호를 기반으로 수정한다.
- 추상화는 필요해질 때 도입한다.
- 미래를 예측한 설계보다 변화 가능한 구조를 우선한다.
- 코드와 경험을 통해 기준을 계속 수정한다.

---

## 구조 진화 기준

아래 상황은 구조 진화 신호로 본다.

- 같은 수정이 반복된다.
- 특정 객체 책임이 과도하게 증가한다.
- 조건문이 계속 증가한다.
- 흐름보다 구조 설명 비용이 커진다.
- 테스트 작성이 어려워진다.
- 변경 영향 범위 예측이 어려워진다.

---

## 추상화 기준

추상화는 아래 조건이 충분히 반복될 때 도입한다.

- 같은 흐름이 반복된다.
- 변경 이유가 동일하다.
- 공통 의미가 자연스럽다.
- 실제 유지 비용이 증가한다.

단순 재사용 가능성만으로는 추상화 이유가 부족하다.

---

## 현재 우선순위

프로젝트는 아래 순서를 우선한다.

1. 명확한 구조
2. 책임 분리
3. 테스트 가능성
4. 유지보수성
5. 재사용성
6. 확장성

---

## 구조보다 문제 우선

아키텍처 패턴 적용 자체를 목표로 삼지 않는다.

예시:

- Hexagonal Architecture
- Clean Architecture
- DDD
- CQRS
- Event Driven

같은 구조도:
- 현재 문제
- 프로젝트 규모
- 유지 비용

을 기준으로 판단한다.

---

## 작은 구조 변경 우선

구조 변경은 가능한 작은 단위로 수행한다.

좋은 예시:

- 책임 하나 이동
- Validator 분리
- Policy 분리
- 패키지 재구성

지양하는 예시:

- 전체 구조 재작성
- 패턴 전체 도입
- 대규모 리아키텍처

---

## 보류 가능한 결정

아래 상황은 결정을 보류할 수 있다.

- 아직 사용 사례가 부족하다.
- 실제 문제 신호가 없다.
- 프로젝트 규모가 작다.
- 구조 비용이 더 크다.

보류는 미완성이 아니라,
판단 시점을 뒤로 미루는 선택으로 본다.

---

## 코드 우선 원칙

문서보다 코드가 우선이다.

코드와 문서가 충돌하면:
- 코드가 잘못된 것인지
- 문서가 오래된 것인지

다시 판단한다.

---

## 마지막 원칙

좋은 구조는:
- 패턴 이름이 많은 구조
가 아니라,
- 수정이 자연스러운 구조
라고 판단한다.
