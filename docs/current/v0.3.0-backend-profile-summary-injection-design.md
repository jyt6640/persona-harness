# v0.3.0 Backend Profile Summary Injection Design

## Goal

Add a narrow backend-only profile summary injection candidate for projects that already have `.persona/project-profile.jsonc`.

The summary should help the agent propose an architecture/technology plan before implementation. It must not become rule enforcement, generated app product-quality certification, frontend/infra productization, or a replacement for `.persona/rules`.

## Evidence

The current `ph intake` path has two positive smokes:

- `docs/evidence-reviews/v0.3.0-intake-planning-smoke.md`
- `docs/evidence-reviews/v0.3.0-intake-planned-implementation-smoke.md`

The planned implementation smoke showed that explicitly reading `.persona/project-profile.jsonc` before implementation helped the model produce:

- domain-first package structure;
- domain repository ports with infrastructure implementations;
- class-based domain entities with behavior;
- strict DTO boundaries;
- no Service-owned storage or id sequence;
- passing `gradle test`, `gradle build`, and manual HTTP smoke.

The remaining gap is that the profile only influenced the run because the prompt explicitly told OpenCode to read it.

## Decision

Proceed with a narrow backend-only profile summary injection design.

The implementation adds a small summary block to Persona Harness injection when all of these are true:

- `.persona/project-profile.jsonc` exists in the project root;
- the profile schema is `persona.project-profile.v1`;
- the profile scope is backend Java/Spring Clean Code;
- the target is a backend bootstrap or Java/Spring target already eligible for Persona Harness injection.

If the profile is missing, malformed, incomplete, frontend/infra scoped, or outside the Java backend MVP, injection should continue without a profile summary.

## Summary Content

The summary should include only project-shaping answers that are already present in the profile.

Recommended fields:

- project context;
- project scale;
- storage style;
- persistence technology;
- migration style;
- package style;
- DTO strictness;
- philosophy overlay selection.

Recommended rendering:

```text
프로젝트 프로필 요약:
- project-context: personal
- project-scale: small
- storage: in-memory
- persistence-technology: undecided
- migration-style: none
- package-style: domain-first
- dto-strictness: strict
- philosophy-overlay: none

프로필 사용 원칙:
- 이 요약은 구현 전 architecture/technology plan 참고용이다.
- 사용자의 README/요구사항과 명시 지시가 우선한다.
- 이 요약은 rule enforcement나 product-quality 보증이 아니다.
```

Do not include unresolved `null` answers as strong guidance. Missing answers may be omitted or rendered as `unanswered`, but they must not be converted into defaults that look like user decisions.

## Injection Placement

Place the profile summary after the file role and before selected rules, or after selected skills and before policies.

Preferred order:

1. current file;
2. file role;
3. project profile summary, if available;
4. selected rules;
5. selected skills;
6. applicable policies;
7. cautions.

This keeps the profile visible as planning context while preserving `.persona/rules` as the actual policy source.

## Non-Goals

- no rule enforcement;
- no build/test failure gate;
- no product-quality certification;
- no frontend/infra/profile-aware productization;
- no company/personal philosophy file loading yet;
- no interactive questionnaire;
- no DB/JPA/Flyway hardcoding;
- no TDD workflow implementation;
- no AST/linter/Guard.

## Implementation Shape

Next implementation should stay small:

- add a profile reader/parser for `.persona/project-profile.jsonc`;
- tolerate missing or malformed profile files without throwing from injection;
- return a diagnostics-style result or optional summary string;
- append the summary to `createInjectionBlock` only for Java backend/bootstrap targets;
- preserve existing selected rules, evidence metadata, and shared-skill routing.

The parser can stay JSONC-light for MVP by using the repo's existing JSONC handling pattern if available. It should not introduce a new dependency.

## Test Criteria

Implementation tests should cover:

- no profile file keeps the current injection block unchanged except for no summary;
- filled backend profile adds a project profile summary;
- unanswered/null fields do not become fake decisions;
- malformed profile does not crash injection;
- frontend/infra or unsupported scope does not inject a backend profile summary;
- Java service target can receive the summary;
- README/bootstrap target can receive the summary;
- non-Java/non-bootstrap target does not receive backend profile summary;
- existing rule selection and shared-skill selection tests still pass.

## Manual QA

After implementation, run a clean project smoke:

1. install Persona Harness locally or from tarball;
2. run `npx ph init`;
3. run `npx ph intake`;
4. fill `.persona/project-profile.jsonc`;
5. ask OpenCode to implement a new Java/Spring Gradle fixture without explicitly saying "read `.persona/project-profile.jsonc`";
6. confirm the model's first plan reflects the profile summary;
7. confirm generated code does not regress on Gradle-only, repository ports, class-based domain behavior, DTO boundary, and Service-owned storage/id sequence.

Observed smoke:

- `docs/evidence-reviews/v0.3.0-profile-summary-injection-smoke.md`

The observed smoke confirmed plan influence without explicitly telling OpenCode to read `.persona/project-profile.jsonc`. It did not run full implementation.

## Risks

- The profile summary can look like a hard rule if the caution text is too weak.
- Unanswered fields can create fake certainty if rendered as defaults.
- Automatic injection can hide whether the model actually read the profile file.
- Adding profile awareness before philosophy files exist can be mistaken for personal/company philosophy support.

## Next Loop

Create a minimal `.persona/workflow/plan.md` artifact path for the `blackbear` planning role.

Do not add frontend/infra profile routing, philosophy overlay loading, AST/linter enforcement, or TDD workflow in that loop.
