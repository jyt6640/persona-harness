# v0.3.0 Interactive Intake Design

## Goal

Design a backend-only interactive intake surface for Persona Harness.

The goal is to make `ph intake` feel closer to a first project conversation:

```text
사용자가 만들고 싶은 서비스를 말한다
-> Persona Harness가 backend project 조건을 한 번에 하나씩 묻는다
-> 답변을 `.persona/project-profile.jsonc`에 구조화해서 저장한다
-> `ph plan`이 답변을 architecture/technology plan에 반영한다
-> 사용자가 계획을 확인한 뒤 구현으로 넘어간다
```

This design drove the first minimal CLI implementation.

## Framing

Interactive intake is planning context, not rule enforcement.

It should help an agent choose a sensible backend architecture and technology line before implementation. It must not claim generated app product quality, enforce a stack, or widen the current MVP into frontend/infra/productized multi-domain support.

The first MVP surface is terminal-based, but the same question flow should also be usable in agent/chat environments. The important behavior is not the UI. The important behavior is:

- ask one question at a time;
- resolve the answer into a stable profile value;
- write the profile only after all required answers are complete;
- keep the profile readable by `ph plan` and profile summary injection.

## Current Problem

`ph intake` currently creates an editable draft file:

```bash
npx ph intake
```

Output:

```text
.persona/project-profile.jsonc
```

The draft is useful, but the user still has to open JSONC and manually fill `questions[].answer`.

That does not match the desired product behavior:

```text
"웹 서비스 만들고 싶어"
-> 하네스가 필요한 backend 질문을 한다
-> 사용자가 답한다
-> 그 답이 계획에 반영된다
```

## Proposed Command Surface

Keep the existing non-interactive path:

```bash
npx ph intake
```

Add a narrow interactive path:

```bash
npx ph intake --interactive
```

Optional aliases can be considered later, but this draft keeps only the explicit flag.

## CLI Behavior

### Existing Profile Missing

If `.persona/project-profile.jsonc` does not exist:

1. Create `.persona/` if needed.
2. Ask backend-only project-shaping questions.
3. Keep answers in memory while the interview is in progress.
4. Write `.persona/project-profile.jsonc` only after all required questions are complete.
5. Print next steps:

```text
Next:
- Review .persona/project-profile.jsonc
- Run npx ph plan
- Ask the agent to read README.md and .persona/workflow/plan.md before implementation
```

### Existing Profile Present

If `.persona/project-profile.jsonc` exists:

- default behavior: do not overwrite;
- print current path and explain safe options;
- future option: `--force` replaces the profile;
- future option: `--merge` can be considered later, but not in MVP.

Draft behavior:

```text
.persona/project-profile.jsonc already exists.
Use --force only if you want to replace the current draft.
```

### Selection Input

The terminal flow should use numeric choices.

Rules:

- number input selects that option;
- `Enter` selects the recommended option for the current question;
- `추천` selects the recommended option for the current question;
- `recommend` selects the recommended option for the current question;
- `미정` stores `undecided` when the question supports uncertainty;
- invalid input re-prompts the same question;
- aborting midway writes no partial profile.

MVP stores only the resolved stable value. It does not store `recommended: true`.

Later UX/debug metadata such as `source: recommended` can be considered, but it is not part of this MVP.

### Non-TTY Environment

If interactive stdin is unavailable:

- fail clearly;
- do not create a partial profile;
- recommend the existing non-interactive path.

Draft message:

```text
Interactive intake requires a TTY. Use `npx ph intake` to create an editable draft.
```

## Question Principles

The interactive flow should be short.

Rules:

- ask backend project-shaping questions only;
- keep the core flow to 8 required groups plus one optional free-text note;
- ask migration only when persistence is needed;
- every question with uncertainty should support `undecided` or `recommend`;
- do not turn empty/skipped answers into hidden defaults;
- do not ask frontend/infra/deployment questions yet;
- do not ask philosophy overlay questions in this intake;
- explain recommendations as suggestions, not rules;
- store user answers exactly enough for `ph plan` and profile summary injection.

## Question Flow

The recommended final flow:

1. Project context
2. Project goal
3. Scale/lifecycle
4. Application type
5. Storage style
6. Persistence technology
7. Package/architecture style
8. Boundary strictness
9. Migration style, conditional when persistence is needed
10. Optional project notes

The “8 required groups” count treats package/architecture style as one required group even though it may ask two related prompts: package split and internal architecture depth.

## Required Questions

### 1. Project Context

Purpose:

- understand where and why the project is used;
- avoid treating learning, solo, company, and open-source projects as the same planning context.

Prompt:

```text
이 프로젝트는 어떤 맥락에 가깝나요?
```

Choices:

- `solo`: 개인 프로젝트
- `team`: 팀 프로젝트
- `company`: 회사/조직 프로젝트
- `open-source`: 오픈소스 프로젝트
- `learning`: 학습 목적
- `recommend`: 추천값 사용

Recommended value:

```text
solo
```

Stored id:

```text
project-context
```

### 2. Project Goal

Purpose:

- separate project purpose from project scale;
- keep `prototype` out of the scale question.

Prompt:

```text
이 프로젝트의 목적은 어느 쪽에 가깝나요?
```

Choices:

- `prototype`: 빠른 검증
- `production-service`: 실제 운영 서비스
- `study`: 학습/미션
- `internal-tool`: 내부 도구
- `portfolio`: 포트폴리오
- `recommend`: 추천값 사용

Recommended value:

```text
production-service
```

Stored id:

```text
project-goal
```

### 3. Project Scale / Lifecycle

Purpose:

- decide architecture depth without over-engineering.

Prompt:

```text
프로젝트 규모와 생명주기는 어느 쪽인가요?
```

Choices:

- `throwaway`: 버려도 되는 짧은 실험
- `small`: 작은 서비스
- `medium`: 중간 규모 이상으로 확장 가능
- `long-lived`: 오래 유지보수할 서비스
- `recommend`: 추천값 사용

Recommended value:

```text
small
```

Stored id:

```text
project-scale
```

Locale note:

- Korean/English labels can vary by user language later.
- The stored values should stay stable English identifiers.

### 4. Application Type

Purpose:

- let the plan distinguish REST API, MVC web, batch, library, and mixed backend shapes.

Prompt:

```text
애플리케이션 형태는 어떤 쪽인가요?
```

Choices:

- `rest-api`: REST API
- `mvc-web`: Spring MVC 웹 애플리케이션
- `batch`: batch/job 중심
- `library`: 라이브러리/모듈
- `mixed`: 여러 형태 혼합
- `recommend`: 추천값 사용

Recommended value:

```text
rest-api
```

Stored id:

```text
application-type
```

### 5. Storage Style

Purpose:

- decide whether state is transient, persisted, external, mixed, or absent.

Prompt:

```text
데이터 저장 방식은 어떤 쪽인가요?
```

Choices:

- `none`: 저장소 없음
- `in-memory`: 서버 실행 중만 유지
- `file`: 파일 기반 저장
- `database`: DB 사용
- `external-api`: 외부 API/서비스가 source of truth
- `mixed`: 여러 저장 방식 혼합
- `recommend`: 추천값 사용

Recommended value:

```text
database
```

Stored id:

```text
storage
```

Reason:

- this prevents the agent from silently choosing in-memory when the user expects persistence;
- it also keeps Service-owned storage/id sequence prohibition visible.

### 6. Persistence Technology

Purpose:

- choose the persistence implementation style when storage needs backend persistence.

Prompt:

```text
영속성 기술은 어떤 쪽을 원하나요?
```

Choices:

- `not-needed`: 현재 필요 없음
- `jdbc-template`: JdbcTemplate
- `jpa`: JPA
- `mybatis`: MyBatis
- `custom`: 직접 구현/특수 저장소
- `recommend`: 추천값 사용

Recommended value:

```text
jdbc-template
```

Stored id:

```text
persistence-technology
```

Reason:

- stack choice is project-specific and should not be hardcoded as a universal rule;
- if the user chooses `recommend`, the CLI resolves it to the recommended value before saving.

### 7. Package / Architecture Style

Purpose:

- decide package organization and internal layer depth before code generation.

Prompt 1:

```text
패키지를 무엇 기준으로 나눌까요?
```

Choices:

- `simple-mvc`: 작은 프로젝트용 Controller/Service/Repository 중심
- `layer-first`: presentation/application/domain/infrastructure를 상위 layer로 분리
- `domain-first`: domain/module 아래 presentation/application/domain/infrastructure
- `feature-first`: feature 단위로 묶고 내부에서 필요한 계층 분리
- `recommend`: 추천값 사용

Recommended value:

```text
domain-first
```

Stored id:

```text
package-style
```

Prompt 2:

```text
선택한 패키지 구조 안에서 내부 계층은 어떻게 나눌까요?
```

Choices:

- `simple-layered`: 단순 layer 구조
- `clean-architecture-light`: 가벼운 Clean Architecture
- `hexagonal-light`: 가벼운 port/adapter 구조
- `strict-clean-architecture`: 엄격한 Clean Architecture
- `recommend`: 추천값 사용

Recommended value:

```text
clean-architecture-light
```

Stored id:

```text
architecture-style
```

Reason:

- current preference is domain-first for clean backend code shape;
- internal architecture depth should still follow project size and user choice;
- exact package names should follow project/domain names.

### 8. Boundary Strictness

Purpose:

- decide how strongly to separate request/response, command/result, validation, and layer boundaries.

Prompt:

```text
계층/DTO/검증 경계는 어느 정도로 엄격하게 가져갈까요?
```

Choices:

- `lightweight`: 작은 프로젝트 기준으로 최소 경계만
- `pragmatic`: 외부 request/response는 분리하고 내부는 필요할 때 분리
- `strict`: request/response, command/result, validation, layer boundary를 엄격히 분리
- `recommend`: 추천값 사용

Recommended value:

```text
strict
```

Stored id:

```text
boundary-strictness
```

Reason:

- this is broader than DTO strictness;
- it influences DTOs, layer boundaries, validation, and later testing style;
- strict separation is the current default clean-code preference, but it remains a planning recommendation.

## Conditional Question

### Migration Style

Ask only when persistence is needed.

Ask when:

- `storage` is `database` or `mixed`; or
- `persistence-technology` is not `not-needed`.

Purpose:

- decide how schema changes are represented.

Prompt:

```text
DB schema/migration 방식은 어떤 쪽인가요?
```

Choices:

- `schema.sql`: Spring Boot `schema.sql`
- `flyway`: Flyway
- `liquibase`: Liquibase
- `none`: DB는 쓰지만 migration 도구는 안 씀
- `not-needed`: DB/persistence 자체가 필요 없음
- `recommend`: 추천값 사용

Recommended value:

```text
flyway
```

Stored id:

```text
migration-style
```

Important distinction:

- `not-needed` means DB/persistence itself is not needed.
- `none` means DB is used, but no migration tool is used.

## Optional Final Note

After the structured choices, ask one optional free-text question:

```text
추가로 agent가 계획 전에 꼭 알아야 할 프로젝트 조건이 있나요? 없으면 Enter를 누르세요.
```

Rules:

- empty input is allowed;
- this is not a required question;
- do not store it in `philosophy.project`;
- store it in `notes.project`.

Stored location:

```json
{
  "notes": {
    "project": "..."
  }
}
```

Reason:

- project-specific conditions, forbidden technologies, testing expectations, and deployment constraints are general project notes;
- `philosophy.project` is too heavy and belongs to a later personalization/policy feature.

## Philosophy / Policy Handling

Do not ask philosophy overlay questions in this intake.

Rationale:

- philosophy/policy is a separate future personalization feature;
- current intake should only shape the backend project plan;
- asking “will you use philosophy later?” adds noise to project intake.

Future commands can be considered separately:

```bash
ph philosophy init
ph policy add
ph rules import
```

Future priority model:

```text
project-specific rule
> company/team rule
> personal preference
> clean-code baseline
> framework default
```

This priority model is not implemented by the current intake.

## Answer Model

Current profile questions use:

```ts
type ProjectProfileQuestion = {
  readonly id: string
  readonly prompt: string
  readonly choices: readonly string[]
  readonly answer: null
}
```

Interactive intake needs the same shape with answered values:

```ts
type ProjectProfileQuestion = {
  readonly id: string
  readonly prompt: string
  readonly choices: readonly string[]
  readonly answer: string | null
}
```

Rules:

- chosen option is stored as a stable string from `choices`;
- `recommend`, `추천`, and empty input are resolved before save;
- MVP stores only the resolved chosen value, not `recommended: true`;
- `undecided` is a real user answer when supported;
- `undecided` is not the same as skipped/null;
- invalid typed input should re-prompt;
- free text is stored in `notes.project`, not in `questions[].answer`;
- aborting midway writes no partial profile.

## Recommendation Behavior

Interactive intake can show recommendations, but the saved answer must still be a resolved stable value.

Examples:

- If the user presses `Enter`, save the current question's recommended value.
- If the user types `추천`, save the current question's recommended value.
- If the user types `recommend`, save the current question's recommended value.
- If `storage=database`, ask migration style and recommend `flyway`.
- If `storage=none`, skip migration and store no migration question or store `not-needed`, depending on implementation compatibility.
- If the user types `미정`, store `undecided` only for questions that explicitly support uncertainty.

MVP should prefer explicit recommended values over storing `recommend` as the answer.

## Plan Integration

`ph plan` should not need a new schema for this draft.

Expected behavior:

- read `.persona/project-profile.jsonc`;
- summarize answered questions only;
- show `undecided` answers as user uncertainty;
- omit skipped `null` answers from strong guidance;
- include `notes.project` as planning context, not as a hard rule;
- keep the warning that profile is planning context, not enforcement.

Example plan summary:

```text
프로젝트 프로필 요약:
- project-context: team
- project-goal: production-service
- project-scale: medium
- application-type: rest-api
- storage: database
- persistence-technology: jdbc-template
- migration-style: flyway
- package-style: domain-first
- architecture-style: clean-architecture-light
- boundary-strictness: strict
- notes.project: 관리자 기능은 이번 범위에서 제외한다.

프로필 사용 원칙:
- 이 요약은 구현 전 architecture/technology plan 참고용이다.
- 사용자의 README/요구사항과 명시 지시가 우선한다.
- 이 요약은 rule enforcement나 product-quality 보증이 아니다.
```

## Non-Goals

- no frontend intake yet;
- no infra/deployment intake yet;
- no company/personal philosophy file loading yet;
- no philosophy overlay question in project intake;
- no automatic philosophy merge;
- no rule enforcement;
- no product-quality certification;
- no AST/linter/Guard;
- no autonomous workflow agents;
- no TDD workflow implementation;
- no default DB/JPA/Flyway hardcoding.

## Implementation Criteria

If this design becomes implementation, the minimal implementation should:

- add `ph intake --interactive`;
- keep `ph intake` non-interactive behavior unchanged;
- keep `--force` overwrite semantics explicit;
- reject unknown flags clearly;
- avoid new dependencies unless Node built-ins are insufficient;
- support TTY input;
- fail safely in non-TTY mode;
- keep generated profile schema as `persona.project-profile.v1`;
- keep answers in memory until all required questions are complete;
- write no partial profile on abort;
- resolve `Enter`, `추천`, and `recommend` to the recommended stable value;
- store `미정` as `undecided` where supported;
- add `notes.project` for optional free text;
- preserve existing profile summary injection behavior.

## Test Criteria

Unit tests should cover:

- existing `ph intake` still creates a draft with null answers;
- `ph intake --interactive` writes selected answers;
- `Enter` writes the recommended stable value;
- `추천` writes the recommended stable value;
- `recommend` writes the recommended stable value;
- `미정` writes `undecided` where supported;
- abort leaves no partial profile;
- migration is asked only when persistence is needed;
- optional free text is stored in `notes.project`;
- existing profile is not overwritten without `--force`;
- non-TTY interactive mode fails with a clear message;
- unknown options still fail;
- `ph plan` reads interactive answers into the profile summary;
- existing profile summary injection tests still pass.

## Smoke Criteria

Clean project smoke should verify:

```bash
npm install -D /absolute/path/to/persona-harness-*.tgz
npx ph init
npx ph intake --interactive
npx ph plan
npx ph plan --status
```

Then run OpenCode with a planning-only prompt:

```text
README.md와 .persona/workflow/plan.md를 읽고,
구현하지 말고 architecture/technology plan만 완성해줘.
마지막에 IMPLEMENTATION_NOT_STARTED를 출력해줘.
```

Expected observations:

- `.persona/project-profile.jsonc` contains selected answers;
- `.persona/workflow/plan.md` reflects those answers;
- OpenCode does not create implementation files in planning-only smoke;
- plan mentions profile choices as planning context, not hard enforcement;
- no frontend/infra questions are introduced.

Implementation smoke is a later loop:

- ask OpenCode to implement from accepted plan;
- verify Gradle only;
- verify repository port/domain + infrastructure implementation;
- verify class-based domain behavior;
- verify Service does not own storage/id sequence;
- verify `gradle test`, `gradle build`, and where applicable `gradle bootRun`.

## Risks

- Too many prompts make intake feel heavy.
- Recommendations can accidentally become fake defaults.
- `undecided` can be confused with skipped/null.
- Free-text project notes can be over-injected as rules.
- Backend-only scope can accidentally drift into frontend/infra.
- Locale-specific labels can diverge from stable stored values.

## Draft Decision

`ph intake --interactive` is implemented against the transcript fixture.

Recommended loop order:

```text
Loop 1: intake transcript fixture + profile schema 확정 - done
Loop 2: ph intake --interactive 구현 - done
Loop 3: ph init -> ph intake --interactive -> ph plan planning-only smoke - done
Loop 4: OpenCode actual implementation smoke
```

Reason:

- interactive CLI changes are expensive after implementation;
- prompts, recommended values, conditionals, and schema shape should be fixed first;
- planning-only smoke should run before actual implementation smoke so failure causes stay separable.
