# v0.3.0 Profile Schema Decision

## Goal

Freeze the `persona.project-profile.v1` shape used by `ph intake`, `ph intake --interactive`, profile summary injection, and `ph plan`.

## Decision

Keep the schema name as:

```text
persona.project-profile.v1
```

Update the v1 question set to the backend-only intake model:

```text
project-context
project-goal
project-scale
application-type
storage
persistence-technology
migration-style
package-style
architecture-style
boundary-strictness
```

Add lightweight project notes:

```json
{
  "notes": {
    "project": null
  }
}
```

Do not add a `philosophy-overlay` intake question.

## Why Keep v1

The current profile schema has not been publicly stabilized as a long-lived external contract. The project is still in local/tarball validation before public npm publish.

Keeping `persona.project-profile.v1` is acceptable because:

- the profile is local project metadata;
- old generated drafts are not a shipped public contract;
- `ph intake --force` can regenerate the draft;
- the reader already treats malformed or unsupported profiles as non-crashing optional context.

If profile compatibility becomes public, future incompatible changes should move to `persona.project-profile.v2`.

## Required Fields

Top-level fields:

- `schema`
- `status`
- `scope`
- `defaults`
- `questions`
- `notes`
- `philosophy`
- `next`

Scope:

```json
{
  "role": "backend",
  "mvp": "java-spring-clean-code",
  "productized": false
}
```

Defaults:

```json
{
  "language": "java",
  "framework": "spring",
  "buildTool": "gradle",
  "testPolicy": "deferred"
}
```

Notes:

```json
{
  "project": null
}
```

Philosophy remains a placeholder/future field:

```json
{
  "company": null,
  "personal": null,
  "project": null,
  "priority": ["project-specific", "company/team", "personal", "clean-code-baseline", "framework-default"]
}
```

## Question Contract

Each question keeps the existing shape:

```ts
type ProjectProfileQuestion = {
  readonly id: string
  readonly prompt: string
  readonly choices: readonly string[]
  readonly answer: string | null
}
```

Non-interactive `ph intake` writes `answer: null`.

Interactive `ph intake --interactive` writes resolved stable values.

## Recommended Values

Recommended values are implementation behavior, not serialized metadata.

The interactive CLI should resolve these inputs before writing:

- `Enter`
- `추천`
- `recommend`

The saved profile should contain the stable answer value, not:

```json
{
  "recommended": true
}
```

## Undecided Values

`미정` should store `undecided` only for questions that explicitly support uncertainty.

MVP decision:

- the non-interactive draft does not include `undecided` in every choice;
- the interactive implementation may add `undecided` as a typed escape when useful;
- `undecided` is distinct from `null`;
- `null` means no answer has been supplied yet.

## Notes Semantics

`notes.project` stores project-specific planning context such as:

- excluded features;
- forbidden technologies;
- temporary constraints;
- deployment assumptions;
- testing expectations.

It is not a rule and must not be injected as enforcement.

## Profile Summary Order

Profile summary injection and `ph plan` should render answered fields in this order:

1. `project-context`
2. `project-goal`
3. `project-scale`
4. `application-type`
5. `storage`
6. `persistence-technology`
7. `migration-style`
8. `package-style`
9. `architecture-style`
10. `boundary-strictness`
11. `notes.project`

Null answers are omitted.

If all answers and notes are empty, render:

```text
- 응답된 항목 없음
```

## Non-Goals

- no frontend intake schema;
- no infra/deployment intake schema;
- no philosophy/policy loading;
- no company/team/personal rule merge;
- no rule enforcement;
- no product-quality certification;
- no AST/linter/Guard.
