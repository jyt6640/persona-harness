# v0.3.0 Workflow Next Surface Decision

## Context

The v0.3.0 workflow track now has enough smoke evidence for the current artifact-first flow:

- `ph intake` creates `.persona/project-profile.jsonc`;
- backend profile summary injection can influence planning without forcing enforcement;
- `ph plan` creates `.persona/workflow/plan.md`;
- `ph plan --status`, `ph plan --accept`, and `ph plan --revise` leave an explicit plan state;
- OpenCode can fill the plan, implement from it, fill implementation/review report templates, and archive workflow artifacts with `ph history`;
- installed package fixture evidence noise is narrowed;
- the Gradle/Spring build-line gap is currently positive after the follow-up smoke.

The next question is not whether another clean generation can pass. It is which workflow surface should become the next narrow productization target.

## Options

### A. Interactive Intake Refinement

Focus:

- make the project-shaping intake feel closer to a real first conversation;
- guide backend users through project context, scale, storage, persistence, migration, package style, and DTO strictness;
- keep answers as planning context for architecture/technology planning;
- avoid turning choices into enforcement.

Why it matters:

- this is closest to the long-term goal where a user says "I want to make a web service" and Persona Harness asks the minimum useful project questions before planning;
- current `ph intake` is a draft file generator, not an interaction surface;
- generated code quality now depends less on one more rule and more on whether early project assumptions are captured.

Risks:

- too many questions can make the flow heavy;
- default-looking answers can create fake certainty;
- frontend/infra questions can accidentally widen the current backend MVP scope;
- company/personal philosophy loading can be mistaken as already productized.

### B. Plan-review Ergonomics

Focus:

- improve how a generated plan is reviewed, revised, accepted, and summarized before implementation;
- make `blackbear` planning more product-like.

Why it matters:

- plan acceptance smokes are positive;
- better plan review may reduce implementation drift.

Risks:

- the core gap is now earlier than plan review: the project assumptions going into the plan are still too manual;
- extra status/report features can become another artifact layer without improving the first conversation.

### C. Test -> Feat -> Refactor Workflow Design

Focus:

- begin the longer-term TDD workflow path;
- define the eventual Test -> Feat -> Refactor progression.

Why it matters:

- this is the desired long-term development workflow;
- it can eventually make Persona Harness more than planning and shape guidance.

Risks:

- product-code shape and intake assumptions should be more stable first;
- moving into TDD now may mix test-policy questions with backend architecture questions;
- implementation would be premature without a design loop.

## Decision

Choose A: **Interactive Intake Refinement**.

## Why

The next useful improvement is making project assumptions explicit before planning. The current guidance/rule layer can already produce acceptable Java/Spring backend shape in clean smokes, and the latest build-line issue is closed unless it regresses.

Interactive intake is also the closest next step toward the user's intended product behavior:

```text
사용자가 만들고 싶은 서비스를 말한다
-> Persona Harness가 backend project 조건을 묻는다
-> 답변을 바탕으로 architecture/technology plan을 만든다
-> 사용자가 계획을 확인한다
-> 구현으로 넘어간다
```

This should stay backend-only for the next loop. Frontend technology, infra deployment, company/personal philosophy files, and autonomous workflow agents remain future tracks.

## Narrow Scope For Next Loop

The next loop selected and implemented a backend-only interactive intake surface.

Candidate command shape:

```bash
npx ph intake --interactive
```

or, if implementation is still premature:

```text
docs/current/v0.3.0-interactive-intake-design.md
```

Minimum backend question groups:

- project context: solo / team / company / open-source / learning;
- project goal;
- project scale and lifecycle;
- application type;
- storage style;
- persistence technology;
- conditional migration style;
- package style;
- architecture style;
- boundary strictness;
- optional `notes.project`.

## Non-Goals

- no frontend/infra questionnaire yet;
- no company/personal philosophy file loading yet;
- no rule enforcement;
- no product-quality certification;
- no autonomous `blackbear`, `Charles`, `jaeki`, or `roach` agents;
- no AST/linter/Guard;
- no TDD workflow implementation yet;
- no DB/JPA/Flyway hardcoding.

## Readiness Criteria For The Next Loop

If the next loop implements `ph intake --interactive`, verify:

- the non-interactive `ph intake` path still works;
- existing profile files are not overwritten unless explicit force behavior is used;
- unanswered choices are stored as unanswered, not fake defaults;
- generated `.persona/project-profile.jsonc` remains backend Java/Spring scoped;
- `ph plan` can read the resulting profile summary;
- clean project smoke confirms the first plan reflects the interactive answers without turning them into enforcement.

If the next loop is design-only, it should still produce:

- exact prompts/questions;
- answer model;
- profile JSONC output shape;
- non-goals;
- smoke criteria for implementation.

## Result

Backend-only interactive intake was designed, implemented, and smoke-verified through `ph init -> ph intake --interactive -> ph plan`.
