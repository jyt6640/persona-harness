# v0.3.0 Gradle / Spring Build Guidance

## Goal

Tighten the Java/Spring backend build guidance after the template-fill/history smoke exposed a local build-line workaround.

This is guidance only. It is not a build enforcement gate, AST/linter rule, or generated app product-quality certification.

## Evidence

The clean template-fill/history smoke generated a buildable Equipment Rental API, but it needed build-file repair:

- local JDK was Java 21 while the generated Gradle toolchain initially targeted Java 17;
- Gradle test initially missed the JUnit Platform launcher;
- `bootJar` failed under the generated Gradle 9.4.0 / Spring Boot 3.3.2 combination;
- OpenCode disabled `bootJar` so `gradle build` could pass.

The run still passed `gradle test`, `gradle build`, and HTTP smoke after repair, but disabling `bootJar` is not the desired default for a Spring Boot application.

## Decision

For Java/Spring backend generation, the build guidance should say:

- choose a mutually compatible Spring Boot plugin, Gradle wrapper/launcher, and Java toolchain line;
- pin the Java toolchain instead of relying on the ambient JDK;
- do not mix a very new Gradle launcher with an older Spring Boot plugin and then disable `bootJar` as the default fix;
- include the JUnit Platform launcher when the selected Gradle/JUnit setup needs it;
- for executable Spring Boot applications, keep `gradle test`, `gradle build`, and a basic `gradle bootRun` smoke viable.
