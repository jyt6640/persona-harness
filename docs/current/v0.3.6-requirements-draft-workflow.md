# v0.3.6 Requirements Draft Workflow

## Goal

Stop vague product ideas from jumping straight into implementation.

When a user says something like `TODO 웹 서비스 만들래`, Persona Harness should treat it as a requirements drafting request, not an implementation request. The AI should write a reviewable requirements draft first, ask the user to review it, and only split/implement after the user approves with a phrase such as `진행하자`.

## Problem

The previous workflow handled two cases:

- requirements file exists: `npx ph workflow split README.md`
- requirements were pasted in prompt: `npx ph workflow capture --stdin`, then `npx ph workflow split`

That is useful only after requirements already exist. It does not handle the common first-user sentence:

```text
TODO 웹 서비스 만들래
```

If the AI implements from that sentence, it must guess scope, users, storage, API contract, validation, and delivery slices. That increases domain misunderstanding, missing requirements, and over-implementation.

## Decision

Add a requirements drafting gate before ticket splitting.

New AI-facing flow:

```text
User: TODO 웹 서비스 만들래
AI: npx ph workflow draft --stdin
AI: requirements draft complete; review backlog/questions/assumptions; say 진행하자 to proceed
User: 진행하자
AI: npx ph workflow approve requirements
AI: npx ph workflow split .persona/workflow/requirements/backlog.md
AI: npx ph workflow next
AI: npx ph workflow implement
```

Generated draft artifacts:

```text
.persona/workflow/requirements/backlog.md
.persona/workflow/requirements/questions.md
.persona/workflow/requirements/assumptions.md
```

The draft is not copied from package shared-skills into project `.persona`. The package owns the workflow skill; the target project only stores workflow outputs.

## Commands

```bash
npx ph workflow draft --stdin
npx ph workflow approve requirements
npx ph workflow split .persona/workflow/requirements/backlog.md
npx ph workflow next
npx ph workflow implement
```

`draft --stdin`:

- requires Persona Harness opt-in through `.persona/`;
- stores the original idea in `.persona/workflow/requirements/latest.md`;
- creates draft backlog/questions/assumptions;
- refuses to overwrite existing draft artifacts.

`approve requirements`:

- requires `.persona/workflow/requirements/backlog.md`;
- changes draft artifact status from `draft` to `accepted`;
- does not implement by itself.

## Runtime Routing

The requirements intent router now separates:

- `requirement-drafting`: vague product idea such as `TODO 웹 서비스 만들래`
- `requirement-approval`: approval phrases such as `진행하자`, but only when a draft backlog exists
- `requirement-implementation`: existing README/requirements implementation requests
- `requirement-continuation`: Step continuation or remaining work

The approval route intentionally stays inactive without a draft backlog so ordinary conversation like `진행하자` is not over-routed.

Runtime transform smoke confirmed:

- `TODO 웹 서비스 만들래` injects `requirement-drafting`;
- `진행하자` does not inject without a draft backlog;
- `진행하자` injects `requirement-approval` when a draft backlog exists.

## Non-Goals

- No generated app quality certification
- No AST/linter enforcement
- No automatic implementation before user review
- No full TDD relay
- No frontend/infra/desktop productization
