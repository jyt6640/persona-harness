# v0.3.1 Workflow Diagnostics Surface

## Goal

Make the local alpha easier to verify after install and harder to misuse from a short OpenCode TUI prompt.

## Problem

Recent TUI use showed a useful split:

- Java/Spring Gradle product-code shape was often influenced in the right direction.
- Workflow evidence discipline was weaker: the agent could skip `.persona/workflow/implementation-report.md` and use raw shell instead of `npx ph bearshell`.

That is not primarily a Java Clean Code rule gap. It is a command-surface and workflow reminder gap.

## Added Surface

- `ph workflow check`
  - Reports plan/report/evidence status.
  - Prints the next workflow action.
  - Report-only; not a product-quality gate.

- `ph doctor`
  - Reports local Node/npm/OpenCode and Persona Harness integration state.
  - Checks `.opencode/opencode.json`, plugin path signal, `.persona/harness.jsonc`, `.persona/rules`, `.persona/workflow/plan.md`, and `.persona/evidence`.
  - npm registry status is `unchecked` in this local diagnostic command.

- `ph smoke`
  - Writes `.persona/workflow/smoke-report.md`.
  - Summarizes the current workflow state for external tester reproduction.

- `ph feedback`
  - Writes `.persona/workflow/feedback-report.md`.
  - Gives external testers a stable feedback template.

- `ph review backend-shape`
  - Writes `.persona/workflow/backend-shape-report.md`.
  - Observes Gradle-only shape, layer/package structure, domain repository port, infrastructure repository adapter, service-owned storage/id sequence, domain behavior, DTO boundary, bootJar status, and verification report evidence.

- `ph evidence summary`
  - Writes `.persona/evidence/summary.md`.
  - Summarizes raw evidence by role, target file name, selected rule, and selected shared skill.
  - Keeps raw evidence untouched and gives humans a smaller first-read surface.

- `ph history`
  - Keeps archiving active workflow artifacts without deleting them.
  - Includes `.persona/evidence/summary.md` content in archive `summary.md` when the evidence summary exists.

## Prompt/Plan Adjustment

`ph plan --prompt` and `ph plan --implement` now make the short TUI route more explicit:

- short requests such as "README 보고 구현해줘" should first run `npx ph workflow check` and `npx ph plan --implement`;
- shell verification should prefer `npx ph bearshell`;
- after implementation, the agent should fill the implementation report and run `npx ph plan --report-filled implementation`;
- before the final answer, the agent should fill the review report after manual QA and run `npx ph plan --report-filled review`.

Injection block dieting is intentionally narrow:

- selected shared-skill metadata still records the package reference path;
- model-facing injection text no longer repeats the full shared-skill file path;
- selected rule paths remain visible because they are the current rule-catalog evidence surface.

## Non-Goals

- Not a generated app product-quality certification.
- Not rule enforcement.
- Not AST/linter/Guard.
- Not frontend/infra productization.
- Not a TDD workflow implementation yet.

## Next Candidate

Run one clean TUI smoke after packaging this change:

1. Install the next local/tarball build.
2. Run `ph init -> ph intake --interactive -> ph policy init -> ph plan -> ph plan --accept`.
3. In OpenCode TUI, use a short prompt such as "README 보고 구현해줘".
4. Check whether the agent runs `npx ph workflow check`, `npx ph plan --implement`, `npx ph bearshell`, `npx ph plan --report-filled implementation`, and `npx ph plan --report-filled review`.
5. Run `npx ph evidence summary` and `npx ph history --id <run-id>` after the run to confirm the human summary path is useful.
