# v0.3.2-alpha.3 ON/OFF A/B Review

## Goal

Validate Persona Harness `0.3.2-alpha.3` with three fresh Java/Spring Gradle README fixtures.

Each fixture was run twice with the same short OpenCode prompt:

```text
README.md 보고 구현해줘
```

The ON runs used `npm install -D persona-harness@alpha` and `npx ph init`.
The OFF runs used only `README.md` and `package.json`.

This review compares workflow behavior, requirement adherence, and backend Clean Code uniformity. It is not a generated app product-quality certification.

- Date: 2026-06-23
- Package: `persona-harness@0.3.2-alpha.3`
- Model: `openai/gpt-5.4-mini-fast`

- Fixtures:
  - `cafe-order-api`
  - `study-group-api`
  - `parking-permit-api`

Each README asked for:

- Gradle only, no `pom.xml`
- Java/Spring Boot REST API
- `presentation / application / domain / infrastructure` separation
- Repository interface in `domain`, implementation in `infrastructure`
- Application Service must not own storage state or id sequence
- Domain model should be class-based with behavior, not passive record-only data
- Request/Response DTO boundary

## Structure Comparison

### Cafe Order API

ON generated a domain-first package layout:

- `customer/application/domain/infrastructure/presentation`
- `menu/application/domain/infrastructure/presentation`
- `order/application/domain/infrastructure/presentation`
- `global/exception`
- `application/dto/command`, `application/dto/result`, `presentation/dto/request`, `presentation/dto/response`
- JDBC repositories, Flyway, H2

OFF generated a flatter package layout:

- `application`
- `domain`
- `infrastructure`
- `presentation`
- in-memory repositories
- no tests

Result: ON was more uniform and closer to the intended package flow. OFF followed the broad layer names, but with less domain-by-domain separation and a weaker verification surface.

### Study Group API

ON generated:

- `studygroup/application/domain/infrastructure/presentation`
- `global/exception`
- command/result and request/response DTO packages
- JDBC repositories, Flyway, H2
- an extra participation repository boundary

OFF generated:

- flat `application/domain/infrastructure/presentation`
- in-memory repositories
- presentation DTOs
- integration test

Result: both were acceptable structurally. ON was more consistent with the domain package style and storage abstraction; OFF was simpler and still clean enough for the small fixture.

### Parking Permit API

- `vehicle/application/domain/infrastructure/presentation`
- `parkingzone/application/domain/infrastructure/presentation`
- `parkingpermit/application/domain/infrastructure/presentation`
- `global/exception`, `global/presentation`, `config`

- JDBC repositories, Flyway, H2
- Gradle wrapper

OFF generated:

- flat `application/domain/infrastructure/presentation`

Result: ON was the clearest positive signal. It followed the target package shape closely and completed `ph workflow finish implement` PASS after report corrections.

## Workflow Findings

ON runs reliably entered the workflow rail:

- `npx ph workflow implement` was attempted in all ON runs.
- Missing plan/report artifacts were detected.
- The model then used `npx ph plan --auto-accept`.
- Reports were filled in all ON runs.

But workflow completion is still noisy:

- Cafe ON: `ph workflow finish implement` eventually PASS, but `ph workflow check` still WARN because a raw shell environment probe remained.
- Study ON: code/test/build/HTTP smoke succeeded, but `ph workflow finish implement` did not PASS because raw shell history stayed sticky even after bearshell retries.
- Parking ON: `ph workflow finish implement` PASS after adding the README read range in a format the checker accepted.

This is the strongest non-code finding: workflow finish is useful, but too format-sensitive and history-sensitive for smooth TUI usage.

## Similarity And Uniformity

Same prompt, same README structure, different domains:

- ON outputs were more uniform across the three fixtures:
  - domain package per feature
  - command/result DTOs in application
  - request/response DTOs in presentation
  - JDBC/Flyway/H2 persistence
  - global exception handling
- OFF outputs were also generally clean, but less uniform:
  - flat package layout
  - in-memory repositories
  - less consistent DTO organization
  - one build failure in Cafe due dependency/plugin compatibility

The practical read is not "OFF is bad." The read is: with the architecture expectations already written in README, a strong model can produce decent OFF results, but Persona Harness ON biases the result toward a more consistent backend shape and leaves auditable workflow artifacts.

## Cold Analysis

Positive:

- ON was stronger on structure uniformity and build reliability.
- No ON run generated Maven.
- No ON run put storage state or id sequence fields in Application Service.
- Domain models were class-based in all ON runs.
- Parking ON is a strong representative demo for the current MVP.

Mixed:

- OFF was better than expected because each README already included explicit architecture expectations.
- ON took longer and sometimes got stuck fighting workflow report formatting.
- ON evidence was inconsistent: Cafe had excellent role evidence; Study/Parking had shallow evidence.

Negative:

- `ph workflow finish implement` is still too brittle around raw shell history and README range formatting.
- ON models still read `.persona/rules` directly instead of only following command surfaces.
- The harness encourages stronger shape, but does not yet guarantee full workflow discipline in short TUI requests.

## Decision

`0.3.2-alpha.3` shows a meaningful ON-positive signal for backend Clean Code uniformity, especially package shape and DTO/repository organization.

It is good enough to continue the Java backend MVP path, but the next work should not be another broad A/B loop. The next work should reduce workflow friction:

1. Make `ph workflow finish implement` less brittle or provide a clearer recovery command.
2. Make README range recording automatic from evidence where possible.
3. Make final verification prefer `ph bearshell` without penalizing earlier harmless environment probes.
4. Improve Java role discovery consistency so Study/Parking also produce role follow-up evidence.
