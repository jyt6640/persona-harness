# Persona Harness v0.4 Evaluation Runbook

## Purpose

This runbook turns `docs/current/v0.4-evaluation-plan.md` into operator steps for a future v0.4 external eval.
It is a skeleton for repeatable execution, not an eval result.

Do not run release, publish, tag, push, AST enforcement, linter enforcement, or product-claim expansion from this runbook.

Release prep may mention this runbook only when this file exists in the target HEAD being prepared.

## Preconditions

Before the first run:

1. Freeze `docs/current/v0.4-evaluation-plan.md`.
2. Freeze all fixture files under `docs/current/evaluation-fixtures/`.
3. Freeze static baseline instruction files for AGENTS.md and CLAUDE.md or cursorrules conditions.
4. Record the Persona Harness package/version that will represent PH ON.
5. Record the exact model/provider policy that will be used.
6. Confirm whether the run matrix is full or reduced due budget.

If any frozen input changes, restart the eval round and record a new runbook version.

## Fixture Selection Order

Run fixtures in this order unless HQ records a reason to change it:

1. `docs/current/evaluation-fixtures/backend-api-no-stack.md`
2. `docs/current/evaluation-fixtures/multi-step-backend.md`
3. `docs/current/evaluation-fixtures/ambiguous-idea-first.md`

Rationale:

- start with stack ambiguity;
- then test long multi-step backend requirements;
- then test idea-first ambiguity and overbuild risk.

Do not choose fixture order based on early results.
If a run is invalid, repeat the same fixture and condition before moving on.

## Baseline Condition Preparation

Prepare a clean working directory per fixture, platform, condition, and repetition.
Do not reuse generated files between conditions.

### Plain Prompt

Use only the fixture README or prompt text.
Do not add AGENTS.md, CLAUDE.md, cursorrules, Persona Harness files, hidden harness instructions, or manual stack steering.

Record:

- fixture path and checksum;
- exact prompt text;
- absence of repo instruction files;
- operator notes about any unavoidable platform setup.

### AGENTS.md Baseline

Use the fixture plus one frozen AGENTS.md baseline file.
The AGENTS.md baseline may describe high-level backend expectations, but must not include Persona Harness command knowledge or PH-specific workflow steps.

Record:

- AGENTS.md baseline path and checksum;
- fixture path and checksum;
- exact prompt text;
- whether the agent reported reading AGENTS.md.

### CLAUDE.md Or Cursorrules Baseline

Use the fixture plus one frozen CLAUDE.md or cursorrules baseline file.
Pick one static baseline shape before the eval starts and reuse it unchanged across fixtures.

Record:

- baseline file type;
- baseline file path and checksum;
- fixture path and checksum;
- exact prompt text;
- whether the agent reported reading the baseline file.

### Persona Harness ON

Use Persona Harness ON only through the documented install/init/workflow path for the selected v0.4 package.
Do not add extra manual guidance beyond the frozen eval prompt and documented PH workflow.

Record:

- package name and version;
- install command;
- init/bootstrap command;
- workflow commands executed;
- profile/bootstrap state;
- evidence/report artifact paths.

## Run Metadata Capture

Every run must record this metadata before scoring:

| Field | Required value |
| --- | --- |
| run id | stable id using fixture, platform, condition, and repetition |
| fixture | fixture file path and checksum |
| condition | plain, AGENTS.md, CLAUDE.md/cursorrules, or PH ON |
| repetition | integer starting at 1 |
| platform | Mac or Windows |
| OS version | exact version string |
| provider | provider name |
| model | model id |
| model version | provider version label when available |
| prompt | exact text sent to the agent |
| timeout policy | wall-clock and per-command timeout |
| token/context limit | configured or observed limit |
| Java/JDK | version and distribution |
| Gradle | wrapper version or installed version |
| Node/npm | versions when PH or generated project uses them |
| operator | person running the eval |
| start/end time | timestamps with timezone |
| invalid-run reason | empty unless rerun is required |

Provider limit, timeout, permissions failure, missing local dependency, and operator recovery must be recorded as metadata, not silently corrected.

## Artifact Archive Naming

Archive every run under a deterministic path:

```text
eval-artifacts/v0.4/{platform}/{fixture-id}/{condition-id}/r{repeat}/
```

Use these ids:

| Fixture | `fixture-id` |
| --- | --- |
| No-stack backend API | `backend-api-no-stack` |
| Long multi-step backend | `multi-step-backend` |
| Ambiguous idea-first | `ambiguous-idea-first` |

Use these condition ids:

- `plain`
- `agents`
- `claude-or-cursorrules`
- `ph-on`

Each archive should contain:

- `metadata.json`
- `prompt.txt`
- frozen fixture copy
- frozen baseline file copy, if any
- generated project snapshot or diff bundle
- command logs for build, test, runtime smoke, PH workflow, and backend-shape review where applicable
- `operator-notes.md`
- `metrics.md`
- `review-package/` after anonymization

Do not edit archived artifacts after scoring.
If a correction is needed, create a new run id or add an append-only errata file.

## Metric Recording

Record primary metrics before secondary metrics.
Primary metrics decide the kill-gate.

### Primary Metrics

For every run, record:

| Metric | Value format |
| --- | --- |
| compile/build rate input | PASS / FAIL / NOT RUN / INVALID |
| test pass input | PASS / FAIL / NOT RUN / INVALID |
| runtime smoke input | PASS / FAIL / NOT RUN / NOT RUNNABLE / INVALID |
| stack alignment | 0 / 1 / 2 plus rationale |
| external failure-mode count | integer plus labels |

Failure labels should be concrete:

- wrong stack;
- compile failure;
- test failure;
- runtime smoke failure;
- timeout;
- provider limit;
- permission limit;
- unfinished pending scope;
- workflow dead-end;
- invalid setup.

### Secondary Metrics

For every run where available, record:

| Metric | Value format |
| --- | --- |
| workflow finish PASS | PASS / WARN / FAIL / NOT APPLICABLE |
| backend-shape result | PASS / WARN / FAIL / NOT RUN |
| backend-shape WARN count | integer |
| evidence coverage | PRESENT / PARTIAL / MISSING / NOT APPLICABLE |
| read coverage | PRESENT / PARTIAL / MISSING / NOT APPLICABLE |
| pending-ticket handling | CLEAR / UNCLEAR / MISLEADING / NOT APPLICABLE |

Secondary metrics explain causes.
They must not override a primary metric regression.

## Blind Package Preparation

After operator scoring but before second reviewer scoring:

1. Copy each run into `review-package/`.
2. Replace condition labels with `Run A`, `Run B`, `Run C`, and `Run D` per fixture and platform.
3. Remove PH-specific path names from summaries where practical, but do not alter command outputs needed for evidence.
4. Keep the condition key in a separate sealed file outside the review package.
5. Preserve raw logs so the reviewer can cite exact failures.

Do not ask the second reviewer to infer which run is PH ON.
If anonymization is impossible because command outputs reveal PH, record the limitation explicitly.

## Second Reviewer Handoff

Give the second reviewer:

- the anonymized review package;
- the scoring rubric from `docs/current/v0.4-evaluation-plan.md`;
- the metadata fields with condition labels removed;
- the primary and secondary metric tables with condition labels removed;
- a request to score failures from command evidence only;
- the expected output shape: scored table, evidence snippets, uncertainty notes, and pass/fail/rerun recommendation.

The second reviewer should not see the condition key until after scoring is complete.

## Kill-Gate Calculation Point

Do not calculate the kill-gate after a single impressive run.
Calculate it only after the preregistered matrix or HQ-approved reduced matrix is complete and second reviewer scoring is recorded.

Calculation order:

1. Exclude invalid runs and record why they were invalid.
2. Compare PH ON to every OFF baseline on the same fixture/platform subset.
3. Check compile/build, test, and runtime smoke for regressions.
4. Compare stack alignment against plain prompt and static baselines.
5. Compare external failure-mode count against the strongest OFF baseline aggregate.
6. Check whether PH ON introduced a recurring new failure mode across two or more fixtures.
7. Apply the baseline kill-gate from `docs/current/v0.4-evaluation-plan.md`.

If PH ON fails a primary threshold, mark the eval as failed even when secondary metrics improve.
If the matrix is incomplete, report "insufficient matrix" instead of pass/fail.

## Reporting Skeleton

Every eval report should include:

- matrix version;
- fixture checksums;
- baseline file checksums;
- PH package/version;
- model/provider/version;
- platform matrix;
- run count by fixture/condition/platform;
- invalid runs and rerun decisions;
- primary metric table;
- secondary metric table;
- second reviewer summary;
- kill-gate result;
- allowed claim and disallowed claim;
- follow-up handoff items.

Allowed claim language:

- "PH ON showed a stack-steering or failure-mode signal under this preregistered matrix."

Disallowed claim language:

- "PH proves generated app quality."
- "Evidence proves model understanding."
- "A/B proves ON is better" without the completed matrix and blind/second reviewer protocol.
