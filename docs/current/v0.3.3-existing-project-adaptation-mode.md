# v0.3.3 Existing Project Adaptation Mode

## Goal

Persona Harness의 코드 균일성 목표를 두 경우로 분리한다.

- 새 프로젝트: `0% -> 80%`까지 Java/Spring backend baseline이 균일하게 나오도록 유도한다.
- 기존 프로젝트: greenfield baseline을 강제하지 않고 기존 코드의 package, layer, naming, repository, DTO, domain style을 먼저 따른다.

## Decision

`ph plan`은 project mode를 plan artifact에 남긴다.

- `greenfield`: `src/main/java`와 `src/test/java`에 Java source가 없을 때.
- `existing-code`: Java source가 이미 있을 때.

`existing-code` mode에서는 다음 원칙을 plan에 명시한다.

> existing code wins over greenfield guidance

즉, 기존 코드에 이미 `web/service/repository` 또는 다른 package/layer convention이 있으면, `presentation/application/domain/infrastructure` baseline을 그대로 밀어붙이지 않는다. 먼저 기존 구조를 읽고, 충돌이 있으면 plan에 남긴 뒤 현재 프로젝트 흐름에 맞춘다.

## Scope

이 구현은 최소 문자열/파일 기반 project-mode observation이다.

- Java source file 존재 여부 확인
- package 선언에서 공통 package root 추정
- 경로 segment에서 layer/style hint 추정
- plan/prompt/workflow rail에 advisory guidance 출력

## Non-Goals

- AST/linter/enforcement gate 아님
- generated app product-quality 보증 아님
- 기존 코드 품질 판정 아님
- 모든 architecture style을 자동 분류하지 않음
- frontend/infra/profile-aware 확장 아님

## Workflow Impact

`ph plan`, `ph plan --prompt`, `ph plan --implement`, `ph workflow implement`는 이제 다음 의도를 더 명확히 전달한다.

- 기존 Java/Spring 코드가 있으면 먼저 existing-code discovery를 수행한다.
- 기존 프로젝트에서는 현재 코드 흐름이 greenfield guidance보다 우선한다.
- 새 프로젝트에서는 Java/Spring backend greenfield baseline을 기본값으로 사용한다.

## Java Role Discovery

Existing-code mode에서 Java role discovery는 두 층으로 나뉜다.

- Evidence: 발견된 Java 파일 전체를 `role-discovery` evidence로 남긴다.
- Follow-up: 모델에게는 role별 대표 Java 파일 1개씩만 read하도록 요청한다.

이렇게 하면 Controller/Service/Repository/DTO/Domain 역할별 injection은 균일하게 model input에 들어가면서, 중복 repository adapter나 부수 파일을 모두 읽느라 workflow가 늘어지는 문제를 줄일 수 있다.

## Validation

Added tests cover:

- empty profiled project creates a `greenfield` project mode plan;
- non-empty Java project creates an `existing-code` project mode plan;
- plan-only and implementation prompts mention existing-code adaptation;
- workflow implementation rail mentions existing-code discovery;
- workflow finish can infer README read coverage from Persona evidence when report ranges are missing.
- existing Spring-style fixture role discovery records Controller, Service, Repository, DTO, Domain, Exception, Test evidence;
- Java Role Read Follow-up asks for representative role files instead of every discovered Java file.
