# Persona Harness v0.4 Evaluation Plan

## Goal

Before v0.4, Persona Harness evaluation must avoid self-confirming claims.
This plan preregisters fixtures, baseline conditions, metrics, thresholds, and reviewer rubric before any v0.4 external smoke run.

This is a QA/eval plan, not an implementation plan.
Do not add AST, linter, enforcement, runtime, CLI, release, publish, tag, or push work from this document.

## Fixture Matrix

Use these three README fixtures as the first v0.4 evaluation set:

| Fixture | Draft path | Purpose | Expected ambiguity |
| --- | --- | --- | --- |
| No-stack backend API | `docs/current/evaluation-fixtures/backend-api-no-stack.md` | Tests whether the condition can steer backend stack without the README naming Java, Spring, Gradle, Maven, or a package layout. | High stack ambiguity |
| Long multi-step backend | `docs/current/evaluation-fixtures/multi-step-backend.md` | Tests long requirement handling, task continuity, and finish behavior over several slices. | Medium implementation scope ambiguity |
| Ambiguous idea-first | `docs/current/evaluation-fixtures/ambiguous-idea-first.md` | Tests whether the condition drafts/splits or over-implements from a vague product idea. | High product ambiguity |

Freeze fixture text before running v0.4 eval.
If a fixture must change, record a new fixture version and restart the eval round.

## Conditions

Run every fixture under these conditions:

| Condition | Description | Harness state |
| --- | --- | --- |
| Plain prompt | README/prompt only, no repo instruction file beyond the fixture. | OFF |
| AGENTS.md baseline | Fixture plus static AGENTS.md guidance that mirrors high-level backend expectations. | OFF |
| CLAUDE.md or cursorrules baseline | Fixture plus static assistant instruction file for non-Codex-style agent guidance. | OFF |
| Persona Harness ON | `persona-harness@v0.4` installed and initialized through the documented workflow. | ON |

The static baseline files must be written before the eval starts and reused unchanged across fixtures.
They must not include hidden PH-specific command knowledge.

## Platform And Repetition Plan

Separate these axes in the result table:

- platform: Mac and Windows;
- condition: plain prompt, AGENTS.md baseline, CLAUDE.md/cursorrules baseline, Persona Harness ON;
- repetition: at least two runs per fixture per condition per platform where budget allows;
- model/provider: exact provider, model id, model version label if available, timeout policy, and token/context limit.

Minimum pre-v0.4 decision set:

- Mac: all three fixtures, all four conditions, two runs each.
- Windows: all three fixtures, plain prompt and Persona Harness ON, one run each if full matrix is too expensive.

Do not merge Mac and Windows results into one score.
Report each axis separately, then summarize only as a directional signal.

## Primary Metrics

Primary metrics are external outcomes outside Persona Harness internals:

| Metric | Pass definition | Failure examples |
| --- | --- | --- |
| Compile/build rate | declared build command completes successfully without disabling required app packaging. | wrong build tool, incompatible plugin, `bootJar` failure, missing source set |
| Test pass rate | generated project tests pass, or no generated tests exist and the agreed smoke test passes. | failing generated tests, test command unavailable |
| Runtime smoke rate | runnable app starts and one happy path plus one failure path can be exercised. | boot failure, endpoint missing, unhandled 500 for expected failure path |
| Stack alignment | generated stack matches fixture and baseline condition expectations. | Maven when Gradle required by condition, frontend-only output for backend fixture, missing backend project |
| External failure-mode count | count wrong stack, compile failure, test failure, runtime smoke failure, timeout, provider/tool limit, unfinished pending scope. | any run-level failure not explained by fixture ambiguity |

Primary metrics decide the kill-gate.

## Secondary Metrics

Secondary metrics explain workflow behavior but do not prove product quality:

| Metric | Meaning |
| --- | --- |
| `ph workflow finish implement` PASS rate | Workflow completion/compliance signal |
| Backend-shape WARN count | Report-only architectural shape signal |
| Evidence coverage | Read/injection/workflow trace depth, not quality proof |
| Read coverage | Whether README/profile/plan/task-card read traces or report lines exist |
| Continuation/pending-ticket handling | Whether the run stops honestly or claims completion with pending scope |

Secondary metrics can justify follow-up work only when primary metrics do not regress.

## Preregistered Success Threshold

Persona Harness ON passes the v0.4 external eval only if all are true:

1. ON has no lower compile/build pass rate than every OFF baseline condition on the same fixture/platform subset.
2. ON has no lower test pass rate than every OFF baseline condition on the same fixture/platform subset.
3. ON has no lower runtime smoke rate than every OFF baseline condition when the generated app is runnable.
4. ON reduces external failure-mode count by at least 20% versus the strongest OFF baseline aggregate on the same completed matrix.
5. ON improves stack alignment rate by at least 20 percentage points versus plain prompt and is not worse than AGENTS.md or CLAUDE.md/cursorrules baselines.
6. ON must not introduce a new recurring failure mode across two or more fixtures, such as workflow dead-end, provider-limit loop, or build-tool mismatch.

If any primary metric regresses, treat v0.4 as a failed eval even if secondary metrics look better.

## Baseline Kill-Gate

If Persona Harness ON cannot reduce external failure modes or improve stack alignment compared with the strongest static baseline, stop AST/enforcement/product expansion.

Kill-gate consequence:

- do not add AST/linter/enforcement gates;
- do not expand product claims;
- do not publish v0.4 as a quality-improvement release;
- instead, diagnose the strongest baseline and update the eval methodology before another product push.

The only acceptable continuation after a failed kill-gate is QA/research/docs work or a narrow workflow reliability fix explicitly approved by HQ/user.

## Blind / Second Reviewer Rubric

Before review, anonymize run condition labels as `Run A/B/C/D` per fixture where practical.
The reviewer should not know which run is Persona Harness ON until after scoring.

Score each run:

| Category | Scale | Scoring instruction |
| --- | --- | --- |
| Build outcome | PASS / FAIL / NOT RUN | Use command output only. |
| Test outcome | PASS / FAIL / NOT RUN | Use generated project test command only. |
| Runtime smoke | PASS / FAIL / NOT RUN / NOT RUNNABLE | Require one happy path and one expected failure path where runnable. |
| Stack alignment | 0-2 | 0 wrong stack, 1 partial, 2 aligned with fixture/condition. |
| Backend shape | 0-2 | 0 confused boundaries, 1 partial boundaries, 2 clear controller/service/domain/repository/DTO boundaries. |
| Requirement coverage | 0-2 | 0 misses major intent, 1 partial, 2 covers core fixture requirements. |
| Failure modes | count + labels | Count concrete failures; do not collapse them into sentiment. |

Second reviewer output must include:

- scored table;
- evidence snippets for each FAIL/WARN;
- uncertainty notes;
- recommendation: pass kill-gate, fail kill-gate, or rerun due invalid setup.

## Reporting Rules

Allowed claim:

- "Persona Harness ON showed a stack-steering or failure-mode signal under this preregistered matrix."

Disallowed claim:

- "Persona Harness proves generated app quality."
- "Evidence proves the model understood the code."
- "A/B proves ON is better" without the completed matrix and reviewer protocol.

## Stop Rules Before v0.4

HQ must ask the user before publish or stop-gate decisions.
Do not run release, publish, tag, or push from this eval plan.
