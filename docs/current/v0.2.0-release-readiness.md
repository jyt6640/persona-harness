# v0.2.0 Release Readiness

## Goal

Document the public npm publish pre-step for Persona Harness `v0.2.0`.

This is not an npm release note. This readiness record fixes what can be trusted before public npm publish:

- local path install
- tarball install
- Java/Spring backend MVP initialization
- Java/Spring backend injection smoke checks
- clean OpenCode `ph bearshell` behavior smoke

The support posture is `v0.2.0-alpha`; the package version remains `0.2.0`. See `docs/current/v0.2.0-alpha-support-contract.md`.

## Version Meaning

`v0.2.0` means:

- Java/Spring backend Clean Code injection MVP is usable through local install and tarball install.
- `persona-harness init` can prepare a clean target project.
- `ph bearshell` can run bounded local command checks from the installed package.
- OpenCode can receive awareness that points repo/build/test checks toward the Persona Harness `ph` command surface.
- README/requirements/Gradle bootstrap targets can receive Java backend project-start guidance.
- Java target files can receive role-specific injection.
- Java file listing can trigger role discovery and role read follow-up.

`v0.2.0` does not mean:

- public npm package is published.
- generated Spring apps are product-quality certified.
- rule compliance is enforced.
- frontend, infra, desktop, or multi-domain shared-skill productization is complete.

## Publish Scope

Public npm publish is explicitly excluded from this loop.

Excluded:

- `npm publish`
- GitHub release
- version bump
- git tag
- release branch

## Guaranteed By Local Path Install

Command shape:

```bash
npm install -D /absolute/path/to/persona-harness
npx persona-harness init
```

Validated guarantees:

- package installs from a local filesystem path.
- `persona-harness` and `ph` bins resolve through `npx`.
- `persona-harness init` creates `.persona/harness.jsonc`.
- `persona-harness init` creates `.persona/rules/`.
- `persona-harness init` creates or merges `.opencode/opencode.json`.
- `.persona/evidence/` is not copied during init.
- `ph bearshell` can execute bounded CLI checks.

## Guaranteed By Tarball Install

Command shape:

```bash
npm pack
mkdir -p /tmp/persona-harness-clean-check
cd /tmp/persona-harness-clean-check
npm init -y
npm install -D /absolute/path/to/persona-harness-*.tgz
npx persona-harness init
```

Validated guarantees:

- `npm pack` creates `persona-harness-0.2.0.tgz`.
- package installs from the tarball.
- `persona-harness` and `ph` bins resolve through `npx`.
- init creates `.persona/harness.jsonc`, `.persona/rules/`, and `.opencode/opencode.json`.
- init does not create `.persona/evidence/`.
- README-linked install/readiness docs are included in the tarball.

## Guaranteed Runtime Surfaces

The current Java/Spring backend MVP guarantees these runtime surfaces:

- Java/Spring `targetFile` detection.
- fileRole-based injection for Java/Spring files.
- `.persona/rules` loading from the installed project.
- metadata-only evidence under `.persona/evidence/phase0`.
- diagnostics-only rule report through `npm run report:rules`.
- MVP scope check through `npm run check:scope:strict`.
- injection value status check through `npm run check:injection-value`.
- Persona Harness CLI runtime helper through `ph bearshell`.
- Injection-block awareness that tells the agent to prefer `ph bearshell` for repo inspection, CLI smoke tests, and large output checks.
- Java role discovery from Java file listing output.
- role read follow-up for discovered Java files.

## Not Guaranteed

The current readiness does not guarantee:

- generated Spring application product quality.
- test sufficiency.
- rule compliance enforcement.
- Guard/AST/linter checks.
- frontend or infra productization.
- desktop app.
- public npm install from registry.
- stable support contract for third-party users.
- exact package or class naming.
- OMO native sidecar, app-server socket, tmux pane, session-context ranking, or spark-model summarization in `ph bearshell`.

## Known Gaps

- Public npm install is not verified because `npm publish` is intentionally deferred.
- OpenCode clean 0-start run produced README bootstrap evidence first; Java role evidence required a target-file read follow-up run.
- The generated Spring app passed tests, but this is not a product-quality certification.
- The tarball still contains vendored `packages/shared-skills` references. They are not release-facing frontend/infra/desktop productization surfaces in this MVP.
- Installed-project command examples should prefer `npx ph ...`; bare `ph ...` only works when the package bin is already on `PATH`.
- `npm audit` may report transitive package vulnerabilities in target clean projects; this readiness loop does not resolve dependency audit posture.
- Frontend, infra, desktop, AST/linter, enforcement, and profile-aware routing remain outside the release-facing MVP.

## Decision

`v0.2.0` is ready as a local/tarball Java/Spring backend MVP readiness candidate.

Do not publish yet. The next publish-oriented loop should decide npm package metadata, README public install language, and support contract.
