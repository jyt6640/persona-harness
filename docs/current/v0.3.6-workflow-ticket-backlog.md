# v0.3.6 Workflow Ticket Backlog

## Goal

Bring the useful part of the blackBear `next-step` workflow into Persona Harness without importing the full TDD relay.

The immediate problem is long requirement implementation. The source can be a README, a requirements document, or a long prompt pasted into the TUI. A single implementation pass can stop after Step 1 or Step 2, then the next continuation is unclear. Persona Harness now needs a small ticket ledger so the AI agent can ask "what is next?" and archive completed work without overwriting history.

## blackBear Reference

blackBear keeps execution artifacts in a clear chain:

- `next-step/work/<task>/00-task-card.md` for active work
- `next-step/history/<task>/...` for completed work
- `next-step/work/backlog.md` for pending work
- scripts that refuse missing cards, broken artifact order, and history overwrite

The important lesson is not the exact agent names. The important lesson is that workflow state is file-backed and script-checked.

## Persona Decision

Persona Harness uses the same shape under `.persona/workflow`:

- `.persona/workflow/backlog.md`
- `.persona/workflow/work/<ticket>/00-task-card.md`
- `.persona/workflow/history/<ticket>/00-task-card.md`
- `.persona/workflow/requirements/backlog.md` for a reviewable requirements draft
- `.persona/workflow/requirements/questions.md` for open questions before implementation
- `.persona/workflow/requirements/assumptions.md` for draft assumptions
- `.persona/workflow/requirements/latest.md` for prompt-captured requirements

New commands:

```bash
npx ph workflow draft --stdin
npx ph workflow approve requirements
npx ph workflow capture --stdin
npx ph workflow split
npx ph workflow split README.md
npx ph workflow next
npx ph workflow archive step-1
```

`draft --stdin` converts a vague product idea into requirements draft artifacts and stops for user review. It does not create implementation tickets.

`approve requirements` marks the draft accepted after the user says to proceed.

`capture --stdin` saves prompt-pasted requirements to `.persona/workflow/requirements/latest.md`.

`split` parses `Step N` markdown headings from a source and creates one task card per step. With an argument, it uses that file. Without an argument, it uses `.persona/workflow/requirements/latest.md`.

`next` prints the first pending ticket and the task card path.

`archive` moves a completed ticket from `work` to `history` and refuses to overwrite an existing history ticket.

`finish implement` fails while pending workflow tickets remain. This prevents Step 1-only implementations from being reported as complete when Step 2+ tickets still exist.

## Non-Goals

- No generated app quality certification
- No TDD relay yet
- No multi-agent role execution
- No scorecard enforcement
- No automatic implementation
- No broad markdown planner

## Known Limits

- Step parsing is intentionally narrow: headings like `## Step 1. ...`
- Existing backlog is not overwritten by `split`
- History is immutable by default; repeated archive attempts fail
- Ticket completion is still human/AI reported through existing implementation and review reports
- Prompt capture requires the AI or user to pipe the prompt text into `npx ph workflow capture --stdin`
- Drafting is intentionally generic. It creates reviewable requirement slices, not a domain-perfect specification.
- `진행하자` routes to approval only when `.persona/workflow/requirements/backlog.md` exists.

## Next Candidate

After this minimal ledger proves useful, the next small improvement is a continuation rail:

```bash
npx ph workflow continue
npx ph workflow next
```

The AI should read the next task card, implement only that ticket, fill reports, then archive it.
