# v0.3.0 Project Intake / Philosophy Workflow

## Goal

`v0.3.0`의 다음 초점은 Java/Spring backend Clean Code MVP 위에 project intake와 optional philosophy overlay를 얹는 것이다.

이 트랙은 generated app product quality를 보증하지 않는다. 목적은 구현 전에 프로젝트 규모, 팀/개인 맥락, 저장소 선택, architecture depth, DTO/package style, 선택 철학을 드러내서 agent가 먼저 계획을 제안하도록 만드는 것이다.

## Current Surface

최소 실행 표면은 CLI draft 생성이다.

```bash
npx ph intake
```

생성 파일:

```text
.persona/project-profile.jsonc
```

`ph intake`는 아직 interactive questionnaire가 아니다. 현재는 사용자가 답을 채울 수 있는 backend profile draft를 만든다.

## Profile Scope

현재 profile은 Java/Spring backend Clean Code planning surface만 다룬다.

고정 기본값:

- language: Java
- framework: Spring
- build tool: Gradle
- test policy: deferred
- MVP: Java/Spring backend Clean Code

현재 질문:

- 개인 프로젝트인지 팀/회사 프로젝트인지
- 프로젝트 규모와 생명주기
- 저장소 성격
- persistence technology
- migration style
- package style
- DTO strictness
- philosophy overlay 사용 여부

## Philosophy Priority

철학이 있는 경우 우선순위는 다음과 같다.

1. company
2. personal
3. clean-code-baseline

project-specific notes는 별도 입력으로 남길 수 있지만, 현재 MVP에서는 자동 병합/주입하지 않는다.

## Intended Flow

1. 사용자가 clean project에서 `npx ph init`을 실행한다.
2. 사용자가 `npx ph intake`를 실행한다.
3. `.persona/project-profile.jsonc`의 `questions[].answer`를 채운다.
4. Agent는 구현 전에 profile과 README를 읽고 architecture/technology plan을 제안한다.
5. 사용자가 계획을 확인한 뒤 구현으로 넘어간다.

## Non-Goals

- frontend/infra/productized multi-domain routing 아님
- 회사/개인 철학 자동 주입 아님
- rule enforcement 아님
- AST/linter/Guard 아님
- generated app product-quality certification 아님
- TDD workflow 구현 아님
- DB/JPA/Flyway 같은 stack 강제 아님

## Why This Is The Next Track

`v0.2.1` local/tarball validation은 Java/Spring backend Clean Code injection, `ph init`, `ph bearshell`, clean README-only generation review까지 통과했다.

남은 큰 질문은 rule을 더 세게 만드는 것이 아니라, 프로젝트마다 달라져야 하는 선택을 어떻게 agent가 먼저 질문하고 계획으로 고정할지다.

## Readiness Criteria

`v0.3.0` 후보가 되려면 다음이 필요하다.

- `ph intake`가 clean project에서 profile draft를 안전하게 만든다.
- 기존 profile을 기본으로 덮어쓰지 않는다.
- README/local install flow에서 `ph init` 다음 선택 단계로 안내된다.
- Java backend MVP scope check와 injection value check를 깨지 않는다.
- profile 내용은 planning surface에 머물고 injection/enforcement로 오해되지 않는다.
