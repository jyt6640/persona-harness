# v0.3.0 Domain Behavior Guidance Review

## Goal

Review whether the repeated generated-demo weakness around anemic domain records/data holders requires another Java/backend guidance change.

This is a scope decision. It is not a new observer, not enforcement, and not generated app product-quality certification.

## Current Finding

The generated demo synthesis narrowed the strongest repeated weak signal to domain behavior:

> Domain models should not default to anemic records/data holders when they own meaningful state. A domain object should keep field-based decisions and state transitions close to its own fields, for example `isOwnedBy(...)`, `canEnroll()`, `loanTo(...)`, or `returnBook()`.

Current guidance already covers this directly.

## Existing Rule Coverage

`backend.java-backend-bootstrap` already says:

- Domain entity/aggregate should not be a record data holder.
- Field-based rules should be judged by the Domain object.
- Application Service should not pull fields through getter/accessor calls to make the same judgment.
- Examples include `isOwner(name)`, `isReturned()`, and `canLoan()`.

`backend.spring.entity` already says:

- Entity/Aggregate is a class, not a record data holder.
- Records are limited to DTOs and true value objects.
- State changes happen through meaningful methods.
- Getter use should not become the default path for external business judgment.

`backend.spring.service` already says:

- Application Service orchestrates use-case flow.
- Validation and policy judgment should be delegated to Domain, Validator, or Policy responsibilities.
- Domain state changes should be expressed through meaningful domain methods.

## Existing Shared-skill Coverage

The Java shared skill already says:

- Use `class` for identity/lifecycle/behavior.
- Use `record` for transparent immutable data, value objects, DTOs, commands, and queries.
- A domain object is state plus behavior, not a data bag.
- The object should own rules it can judge from its own state.
- Getter-driven business logic is rejected.
- Example rejected shape: `public record Order(Long id, String name, OrderStatus status)`.

The domain-model reference is stronger than a short rule because it explains construction, self-validation, behavior over getters, getter policy, immutability, value objects, first-class collections, and policy extraction.

## Decision

Do not add another wording change in this loop.

Reason:

- The exact target guidance is already present in both `.persona/rules` and the Java shared skill.
- Repeating the same sentence in more files would increase maintenance noise without proving generated behavior improves.
- The next evidence gap is not wording existence. The next evidence gap is whether a fresh clean generation actually follows the existing wording.

## Next Validation

Run one clean Java/Spring Gradle generation with the current `v0.3.0-alpha.1` guidance and evaluate only the domain behavior criterion first.

Recommended fixture:

- a small use case with meaningful domain state and ownership/capacity rules;
- not roomescape/reservation;
- not another broad A/B loop;
- require build/test/manual HTTP smoke only as sanity checks, not as product-quality certification.

Example acceptance evidence:

- domain entities with meaningful state are classes, not records;
- field-based judgment is inside the domain object, such as `isOwnedBy(...)`, `canEnroll()`, `loanTo(...)`, or `returnBook()`;
- Application Service orchestrates and does not duplicate the same field judgment through getters/accessors;
- Repository boundary, DTO boundary, and Service storage/id sequence remain stable.

## Non-goals

- no AST/linter/enforcement gate;
- no frontend/infra/desktop expansion;
- no generated app quality guarantee;
- no package/root naming A/B loop;
- no broad rule rewrite.
