# v0.3.0 Blackbear Plan Artifact

## Goal

Add minimal workflow artifacts for the `blackbear`, `jaeki`, and `roach` roles.

The artifacts make the plan-first workflow concrete without introducing autonomous subagents, rule enforcement, frontend/infra routing, or TDD workflow enforcement.

## Command Surface

```bash
npx ph plan
npx ph plan --status
npx ph plan --accept
npx ph plan --revise
npx ph plan --report-filled implementation
npx ph plan --report-filled review
npx ph history --id completed-run
```

Output:

```text
.persona/workflow/plan.md
.persona/workflow/implementation-report.md
.persona/workflow/review-report.md
```

`ph plan --force` replaces existing workflow drafts.

`ph plan --status` reads the current `Status:` line from `.persona/workflow/plan.md`.

`ph plan --accept` marks the plan as `accepted`.

`ph plan --revise` marks the plan as `needs-revision`.

`ph plan --report-filled implementation` marks `.persona/workflow/implementation-report.md` as `filled` after the report has been filled.

`ph plan --report-filled review` marks `.persona/workflow/review-report.md` as `filled` after the report has been filled.

`ph history` archives completed workflow artifacts into `.persona/workflow/history/<archive-id>/` after a run has been used. It copies present artifacts, records missing artifacts in `summary.md`, and does not delete active workflow files.

## Inputs

The draft is based on local project context:

- `README.md` presence and first heading;
- `.persona/project-profile.jsonc` backend profile summary when present;
- the current Java/Spring backend Clean Code planning boundary.

If README or profile is missing, the command still creates a draft and records the missing or unanswered state. Missing context is not converted into fake defaults.

## Artifact Shape

The generated plan file contains:

- `blackbear` role marker;
- requirements source;
- backend project profile summary;
- architecture/technology plan checklist;
- package/layer plan checklist;
- storage/persistence decision checklist;
- DTO/domain/repository boundary checklist;
- non-goals;
- acceptance note that implementation should not start until the plan is reviewed or accepted.

## Acceptance Status

Plan acceptance is a local workflow marker:

- `draft`: the plan was generated and has not been explicitly accepted;
- `needs-revision`: the plan should be revised before implementation;
- `accepted`: the user or workflow has marked the plan ready for implementation.

This status does not enforce command execution, fail builds, certify generated app quality, or prove plan correctness. It only makes the pre-implementation decision visible in `.persona/workflow/plan.md`.

The generated implementation report template contains:

- `jaeki` role marker;
- `template` status marker;
- implemented file summary slots;
- Gradle verification slots;
- `gradle bootRun` slot for runnable Spring Boot apps;
- HTTP happy path / failure path manual QA evidence slots;
- fallback slot for recording why live manual QA could not run, including stderr or key logs;
- status lifecycle reminder to run `ph plan --report-filled implementation` after filling the report;
- changed decisions and remaining limits slots.

The generated review report template contains:

- `roach` role marker;
- `template` status marker;
- requirements check slots;
- Controller/Service/Repository/DTO/domain boundary review slots;
- Gradle verification review slots, including `gradle bootRun` when the app is runnable;
- HTTP happy path / failure path manual QA evidence review slots;
- status lifecycle reminder to run `ph plan --report-filled review` after filling the report;
- remaining limits slots.

## Report Status Lifecycle

Workflow reports start as:

```text
Status: template
```

After the implementation or review actor fills a report, the status can be marked explicitly:

```bash
npx ph plan --report-filled implementation
npx ph plan --report-filled review
```

This changes only the selected report status to:

```text
Status: filled
```

It does not update plan acceptance, enforce report quality, or certify generated app quality.

## Non-Goals

- no autonomous agent orchestration;
- no `Charles`, `jaeki`, or `roach` autonomous implementation yet;
- no rule enforcement;
- no product-quality certification;
- no frontend/infra workflow;
- no company/personal philosophy file loading;
- no TDD workflow enforcement;
- no AST/linter/Guard.

## Next Loop

Plan-based implementation, repeat workflow, and template-fill/history smokes passed. `ph plan` now creates minimal implementation/review report templates too, and `ph history` archives completed workflow artifacts.

The template-fill smoke confirmed OpenCode filled the pre-created implementation/review report files instead of creating ad hoc report files, then `ph history` archived all three workflow artifacts.

Plan acceptance is now represented by `ph plan --status`, `ph plan --accept`, and `ph plan --revise`.

After the accepted interactive profile smoke, the implementation report template was tightened so generated runnable Spring Boot apps ask for `gradle bootRun` plus live HTTP happy/failure path evidence, or an explicit reason and stderr/key logs when that is not feasible.

The follow-up live HTTP QA OpenCode smoke confirmed the model filled those live HTTP QA slots itself. `ph plan --report-filled implementation|review` now covers the remaining status-label gap without adding an enforcement gate.

Do not implement `Charles`, `jaeki`, or `roach` as autonomous agents until these workflow artifacts stay useful in repeated smokes.
