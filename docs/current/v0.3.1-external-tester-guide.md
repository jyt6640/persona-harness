# v0.3.1 External Tester Guide

## Goal

Validate the published Persona Harness alpha from a clean project, using the same path a new user would use.

This guide checks:

- install friction
- OpenCode plugin connection
- `ph` workflow command surface
- generated Java/Spring Gradle backend shape
- workflow reports and evidence

It does not certify generated application product quality.

## Scope

Use this flow:

```text
npm install -D persona-harness@alpha
npx ph init
npx ph intake
npx ph policy init
npx ph plan
npx ph plan --accept
OpenCode short implementation request
npx ph workflow check
npx ph review backend-shape
```

Do not test frontend, infra, desktop app, AST/linter enforcement, or TDD workflow in this pilot.

## Prerequisites

- Node.js 20+
- npm
- Java 21+
- Gradle
- OpenCode CLI with a configured model provider

Verify OpenCode:

```bash
opencode --version
opencode auth list
```

The current smoke model is:

```text
openai/gpt-5.4-mini-fast
```

Other OpenCode-compatible models can be tested, but record the model in the feedback.

## Clean Project Setup

```bash
mkdir -p /tmp/persona-harness-alpha-check
cd /tmp/persona-harness-alpha-check
npm init -y
npm install -D persona-harness@alpha
npx ph init
npx ph doctor
```

Expected:

- `.opencode/opencode.json` exists
- `.persona/harness.jsonc` exists
- `.persona/rules` exists
- `ph doctor` shows `Persona package version: 0.3.1-alpha.0` or newer
- `ph doctor` shows OpenCode present

## README Fixture

Create `README.md`:

```md
# Equipment Rental API

Java/Spring Boot와 Gradle로 장비 대여 REST API를 구현한다.

## 요구사항

* 장비를 등록할 수 있다.
* 장비 목록을 조회할 수 있다.
* 회원을 등록할 수 있다.
* 회원이 장비를 대여할 수 있다.
* 이미 대여 중인 장비이거나 수량이 부족하면 대여에 실패한다.
* 장비를 대여한 회원 본인만 반납할 수 있다.
* 존재하지 않는 장비, 회원, 대여 요청에는 적절한 예외 응답을 반환한다.

## 기술 조건

* Java 21 이상
* Spring Boot 3.x
* Gradle only
* REST API
* 화면 구현 없음
* 저장소는 단순 in-memory 구현으로 시작해도 된다.
* Repository interface는 domain 패키지에 두고, 구현체는 infrastructure 패키지에 둔다.
* Application Service는 저장소 상태나 id sequence를 직접 소유하지 않는다.
* Domain은 record가 아니라 상태와 행동을 가진 class로 만든다.
* Controller, Service, Domain, Repository, DTO 책임을 분리한다.
```

Avoid the older `reservation`, `roomescape`, or `book loan` fixtures when possible.

## Plan Setup

For the current alpha smoke, a non-interactive profile draft is enough:

```bash
npx ph intake
npx ph policy init
npx ph plan
npx ph plan --accept
npx ph workflow check
```

Expected before implementation:

```text
Workflow status: WARN
plan.md: accepted
implementation-report.md: template
review-report.md: template
```

That WARN is expected because implementation has not happened yet.

## Implementation Smoke

Run OpenCode with a deliberately short user-style prompt:

```bash
opencode run --dir . \
  --model openai/gpt-5.4-mini-fast \
  --dangerously-skip-permissions \
  "README.md 보고 구현해줘"
```

The agent should run or follow:

- `npx ph workflow start implement`
- `npx ph plan --implement`
- `npx ph bearshell` for Gradle verification
- `.persona/workflow/implementation-report.md`
- `npx ph plan --report-filled implementation`
- `.persona/workflow/review-report.md`
- `npx ph plan --report-filled review`
- `npx ph workflow finish implement`

The agent should not read `.persona/rules` directly and should not use installed package fixtures as implementation context.

## Required Verification

Ask the agent to verify, or run manually if needed:

```bash
npx ph bearshell --shell 'gradle test'
npx ph bearshell --shell 'gradle build'
npx ph workflow check
npx ph evidence summary
npx ph review backend-shape
```

For runnable Spring Boot apps, also check one happy path and one failure path over HTTP.

## What Counts As A Useful Positive Signal

- `npm install -D persona-harness@alpha` works.
- `npx ph init` creates `.persona` and `.opencode`.
- OpenCode starts with `ph workflow start implement` or `ph plan --implement`.
- Generated project stays Gradle-only.
- Repository ports are in `domain`.
- Repository implementations are in `infrastructure`.
- Application Services do not own `Map`, `AtomicLong`, `nextId`, or `idCounter`.
- Domain models are classes with behavior, not passive records only.
- Request/response DTO boundaries are visible.
- `gradle test` and `gradle build` pass.
- `ph workflow finish implement` passes.
- `ph review backend-shape` is mostly PASS or all PASS.

## What Does Not Count Alone

- Install succeeded but no OpenCode run was attempted.
- `ph plan` created files but the plan was never accepted.
- Generated code exists but Gradle verification was not attempted.
- Reports are still `Status: template`.
- Code only works after manual edits outside the AI run.
