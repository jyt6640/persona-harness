# v0.3.0 Step API Contract Scope Decision

## Goal

Keep the old roomescape step API contract rules available for the original step fixtures, but prevent them from becoming default Java/Spring clean-project guidance.

## Problem

The generated Java target-role follow-up showed role-specific injection working for generated Controller, Service, Repository, DTO, and domain files. It also exposed a scope bug: `backend/step1-api-contract.md` could be selected for arbitrary clean-project Controller/DTO targets.

That rule says `POST /reservations` must return 200 OK and that 201 Created is wrong, and says `DELETE /reservations/{id}` must return 200 OK and that 204 No Content is wrong. Those statements are correct only for the old roomescape step fixture. They are harmful as general REST guidance for new Java/Spring projects.

## Decision

`backend/step1-api-contract.md` and `backend/step2-3-api-contract.md` are roomescape step fixture rules, not default backend Clean Code rules.

They are now selected only when the target path is under a `roomescape` package/path. Clean Java/Spring targets still receive the general backend Controller/DTO guidance, including the rule that API path, method, status code, request body, and response body follow the current project requirements.

## Implementation Surface

- `.persona/rules/backend/step1-api-contract.md` and `.persona/rules/backend/step2-3-api-contract.md` use roomescape-scoped globs.
- `src/phase0/rule-loader.ts` has an additional target-scope gate so fallback loading cannot inject step contract rules into non-roomescape targets.
- `.persona/rules/backend/spring-controller.md` carries the generic HTTP contract guidance for clean projects.
- `tests/phase0-hooks.test.ts` distinguishes clean Java/Spring targets from roomescape step fixture targets.

## Non-Goals

- This is not a new observer.
- This is not enforcement.
- This does not certify generated app product quality.
- This does not remove the old roomescape step fixture evidence.
- This does not change frontend, infra, desktop, or shared-skill productization scope.

## Verification

Targeted verification:

```bash
npm test -- tests/phase0-hooks.test.ts
```

Expected behavior:

- Clean `com/example` Controller/Test/Request/Response targets do not select `backend/step1-api-contract.md`.
- Roomescape Controller/Test/Request/Response fixture targets keep selecting the matching step contract rule for `step1` or `step2-3`.

## Next

Run a clean Java/Spring workflow smoke after this scope change. The main thing to check is whether generated Controllers can choose REST-appropriate status codes from the README/profile without old roomescape 200 OK pressure.
