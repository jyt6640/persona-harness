# v0.3.0 Philosophy / Policy Overlay Design

## Goal

Design the next backend-only planning surface for optional philosophy and policy overlays.

The overlay should let a project add company/team guidance and personal backend philosophy above the default Clean Code baseline without turning the harness into enforcement, AST/linter, product-quality certification, or frontend/infra productization.

## Context

`ph intake` already captures project-shaping choices:

- project context;
- project goal;
- scale/lifecycle;
- application type;
- storage;
- persistence technology;
- migration style;
- package style;
- architecture style;
- boundary strictness;
- project note.

That profile is useful for architecture/technology planning, but it intentionally does not ask philosophy/policy questions. The next surface should keep that separation.

## Decision

Add a separate philosophy/policy overlay surface after project intake.

Priority inside the overlay plane:

1. company/team policy;
2. personal philosophy;
3. Clean Code baseline;
4. framework default.

README, current user prompt, and accepted workflow plan remain the requirements plane. They decide product behavior, route/status/body contracts, and feature scope. Overlay guidance should shape code style, boundaries, architectural preferences, and implementation taste; it should not silently rewrite functional requirements.

## File Model

Recommended local project files:

```text
.persona/
  policies/
    company/
      backend.md
    personal/
      backend.md
    overlay.jsonc
```

`overlay.jsonc` is the local activation and ordering file:

```jsonc
{
  "schema": "persona.policy-overlay.v1",
  "status": "draft",
  "scope": {
    "role": "backend",
    "mvp": "java-spring-clean-code",
    "productized": false
  },
  "enabled": true,
  "priority": ["company", "personal", "clean-code-baseline"],
  "sources": {
    "company": ".persona/policies/company/backend.md",
    "personal": ".persona/policies/personal/backend.md"
  },
  "limits": {
    "maxBulletsPerSource": 5
  }
}
```

The markdown files stay simple:

```markdown
# Backend Company Policy

- Prefer domain-first packages for production backend services.
- Use repository ports in domain and implementations in infrastructure.
- Use Flyway for schema migration when the project uses a relational database.
```

```markdown
# Backend Personal Philosophy

- Domain models should be classes with behavior, not passive records.
- Application Services orchestrate use cases and should not own storage state.
- Keep request/response DTOs separate from command/result DTOs when boundaries diverge.
```

## Command Surface

Future minimal command:

```bash
ph policy init
```

Recommended behavior:

- create `.persona/policies/overlay.jsonc`;
- create empty `.persona/policies/company/backend.md`;
- create empty `.persona/policies/personal/backend.md`;
- do not overwrite existing policy files without `--force`;
- do not create frontend/infra policies in this loop;
- do not create global user-home policies in this loop.

Future optional variants:

```bash
ph policy status
ph policy init --force
```

Do not add `ph intake` questions for this. Project intake should remain about project shape; policy overlay should be opt-in and file-based.

## Injection Surface

When all conditions are true:

- `.persona/policies/overlay.jsonc` exists;
- `enabled` is true;
- schema is `persona.policy-overlay.v1`;
- scope is backend Java/Spring Clean Code;
- the target is backend bootstrap or Java/Spring target already eligible for Persona Harness injection;
- at least one configured markdown source exists and has bullet policies.

Then render a small policy overlay summary in the injection block.

Preferred order:

1. current file;
2. file role;
3. project profile summary, if available;
4. policy overlay summary, if available;
5. selected rules;
6. selected skills;
7. applicable policies;
8. cautions.

Suggested rendering:

```text
정책/철학 오버레이:
우선순위: company > personal > Clean Code baseline

Company policy:
- Prefer domain-first packages for production backend services.
- Use repository ports in domain and implementations in infrastructure.

Personal philosophy:
- Domain models should be classes with behavior, not passive records.
- Application Services orchestrate use cases and should not own storage state.

오버레이 사용 원칙:
- README, 현재 사용자 지시, accepted plan의 기능 요구사항이 우선한다.
- company policy는 personal philosophy보다 우선한다.
- 이 오버레이는 rule enforcement나 product-quality 보증이 아니다.
```

## Conflict Model

Separate two planes:

1. Requirements plane:
   - current user prompt;
   - README/requirements;
   - accepted `.persona/workflow/plan.md`;
   - `.persona/project-profile.jsonc` project-shaping answers.

2. Policy plane:
   - company policy;
   - personal philosophy;
   - Clean Code baseline;
   - framework default.

Rules:

- Requirements plane wins for product behavior and API contract.
- Company policy wins over personal philosophy when both talk about the same implementation choice.
- Personal philosophy can refine Clean Code baseline when company policy is silent.
- Clean Code baseline fills gaps when no overlay exists.
- If an overlay appears to contradict README behavior, the agent should follow README and mention the conflict in the plan/review report.

## Reader / Parser Shape

Future implementation should stay small:

- JSONC-light parser for `.persona/policies/overlay.jsonc`;
- markdown bullet extraction only;
- diagnostics-only findings for missing files, malformed JSONC, unsupported schema, unsupported scope, and empty sources;
- no new dependency;
- non-crash fallback to no overlay;
- max bullets per source from `overlay.jsonc`, defaulting to 5.

Do not build a full rules engine, priority resolver, AST/linter, or policy compiler.

## Evidence / Report Shape

If implemented, evidence metadata can add:

```json
{
  "selectedPolicyOverlay": {
    "enabled": true,
    "sources": ["company", "personal"],
    "diagnostics": []
  }
}
```

This is metadata-only. It should not become a pass/fail gate.

## Non-Goals

- no enforcement gate;
- no build/test failure on policy mismatch;
- no AST/linter/Guard;
- no frontend/infra policy loading;
- no desktop app;
- no global home-directory personal policy in the MVP loop;
- no automatic company policy discovery from GitHub/org/repo settings;
- no TDD workflow implementation;
- no generated app product-quality certification.

## Test Criteria For Future Implementation

The next implementation loop should test these first:

- no `overlay.jsonc` keeps current injection unchanged except for no overlay summary;
- valid company-only policy renders company bullets;
- valid personal-only policy renders personal bullets;
- company + personal renders in company-first order;
- malformed `overlay.jsonc` does not crash injection and produces diagnostics;
- missing markdown source does not crash injection and produces diagnostics;
- unsupported scope does not inject a backend overlay;
- generated evidence records selected policy overlay metadata without making it a gate.

## Next Loop

Implement the narrow backend-only policy overlay surface:

1. add `ph policy init`;
2. read `.persona/policies/overlay.jsonc` with diagnostics-only fallback;
3. extract a small number of markdown bullets from company and personal backend policy files;
4. inject the overlay summary only as planning context for eligible Java/Spring backend targets;
5. keep frontend/infra/global policy loading, enforcement gates, AST/linter checks, and product-quality claims out of scope.
- non-Java/non-bootstrap targets do not receive backend overlay;
- max bullet limit is respected;
- existing profile summary, selected rules, selected skills, and evidence format remain stable.

## Smoke Criteria For Future Implementation

After implementation:

1. create a clean Java/Spring project;
2. install Persona Harness from local path or tarball;
3. run `ph init`;
4. run `ph intake` or `ph intake --interactive`;
5. run `ph policy init`;
6. fill company and personal backend markdown files with conflicting but harmless implementation preferences;
7. run `ph plan`;
8. ask OpenCode to read README and the plan before implementation;
9. confirm the plan mentions company > personal > Clean Code priority;
10. confirm generated code follows company policy where it conflicts with personal preference;
11. confirm README functional/API contract still wins over overlay preferences.

## Decision

Proceed next with a narrow backend-only `ph policy init` and policy overlay summary implementation.

Keep it report-only and planning-context-only. Do not add enforcement, frontend/infra policy loading, global personal policy discovery, or TDD workflow in that implementation loop.
