## Findings

Good signals:

- Clean tarball install plus `npx ph init` worked in a repo outside Persona Harness.
- README-only bootstrap created a non-trivial Spring Boot Gradle project from zero.
- The generated project used meaningful root package naming: `com.example.librarylending`.
- The generated project avoided Maven.
- Application services did not own `Map`, `List`, `AtomicLong`, `nextId`, `idCounter`, or `sequence`.
- In-memory state and id generation stayed in infrastructure repository implementations.
- Controllers, services, DTOs, domain objects, and repositories were separated.
- The model self-corrected a failing `gradle test`.
- Target-file follow-up improved structural quality, especially repository port placement.
- Live HTTP QA matched the main happy paths and conflict/not-found paths.

Weak signals:

- README-only bootstrap did not place repository interfaces in `domain`; it placed `BookRepository`, `MemberRepository`, and `LoanRepository` in `infrastructure`.
- README-only bootstrap needed a follow-up to reach the preferred Repository boundary shape.
- The generated tests are useful smoke coverage but this review does not evaluate long-term test design quality.
- The app uses in-memory repositories. That was allowed by the fixture, but persistence choice remains outside this review.
- Domain objects are mostly Java records with light behavior. `Loan` owns `isReturned()` and `markReturned(...)`, but `Book` and `Member` are data holders and the active-loan rule is still orchestrated in `LoanApplicationService`.
- Exception messages are user-visible raw English strings. They are adequate for a demo but not a mature API error contract.
- The return endpoint uses `POST /loans/{loanId}/return`; it works, but a real product decision may prefer a clearer command resource or status transition contract.

## Decision

Decision: keep `v0.2.1` as local/tarball usable for Java/Spring backend Clean Code validation, and close the narrow repository/domain-record `v0.2.2` candidate for now.

`v0.2.1` is good enough to continue local generation checks because:

- install/init/plugin/evidence worked,
- README-only bootstrap generated a buildable Gradle Spring app,
- target-file read follow-up moved the first generated code into the expected Clean Code shape,
- the second README-only clean generation produced domain repository ports, infrastructure implementations, class-based domain entities, and domain self-judgment methods without a target-file follow-up,
- final `gradle test` and `gradle build` passed.

The first library-lending pass exposed a real guidance gap. The second course-enrollment pass did not repeat it after the narrow guidance change, so the current evidence does not justify another rule/skill tightening loop.

## v0.2.2 Candidate

Status: closed for now. Reopen only if a future README-only clean generation again places repository ports in `infrastructure` or generates domain entities as record-style data holders.

```text
Repository ports/interfaces belong in the domain package; concrete persistence/storage implementations belong in infrastructure.
Application services depend on the domain repository port, not the infrastructure package.

Domain entities/aggregates are classes, not record data holders.
Rules that can be judged from the object's own fields belong on the domain object as named behavior,
for example isOwner(name), isReturned(), or canLoan().
```

Likely placement:

- backend package-structure rule, or
- Java shared-skill `repository-pattern.md`, or
- backend entity/domain rule, or
- bootstrap guidance, if the miss happens before Java target-file reads.

Do not broaden into DB choice, TDD workflow, frontend/infra, observer work, or AST/linter enforcement from this result.

## Next

Recommended next loop: move to `v0.3.0` project-intake / philosophy workflow planning.

Keep the current Java backend Clean Code guidance stable while planning how a future user/project intake should choose defaults such as project scale, team/personal context, persistence style, package style, and optional personal/company philosophy overlays.
