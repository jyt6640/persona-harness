# v0.3.0 Bearshell Guidance Clean Generation Review

## Goal

Check whether the updated workflow prompt/report templates make OpenCode prefer `npx ph bearshell` and `npx ph ...` in a clean Java/Spring Gradle project.

This review observes command behavior only. It is not a generated app product-quality certification.

## Setup

- Persona Harness package: local tarball `persona-harness-0.3.0-alpha.1.tgz`
- Clean project: `/Users/yongtae/Desktop/persona-harness-artifacts/generated-apps/persona-v030-bearshell-guidance-check`
- Model: `openai/gpt-5.4-mini-fast`
- Prompt asked OpenCode to read `README.md`, `.persona/project-profile.jsonc`, `.persona/policies`, and `.persona/workflow/plan.md`, confirm accepted plan status, implement a Java/Spring Gradle API, prefer `npx ph bearshell`, fill reports, and run `npx ph plan --report-filled ...`.

## Result

PASS with a narrow limitation.

OpenCode used the Persona Harness command surface for Gradle verification, executable boot smoke, and report status lifecycle.

It still used raw shell for two small command surfaces:

- `java -version`
- one chained `curl` HTTP smoke command

That means the behavior is better than the previous run, but not yet identical to OMO `sparkshell` habit formation for every shell command.

## Decision

Keep the current `npx ph bearshell` guidance.

Do not add enforcement or a shell wrapper gate. The remaining raw shell usage is small and does not block the current Java/Spring backend alpha.

## Next

If the goal becomes stronger OMO-like command habit formation, tighten the implementation-report template one more step:

- prefer `npx ph bearshell --shell '<curl command>'` for HTTP smoke;
- ask reports to distinguish ph-wrapped commands from raw commands instead of saying all raw shell was avoided.
