# v0.3.0 Live HTTP QA OpenCode Smoke

## Goal

Verify whether the tightened `.persona/workflow/implementation-report.md` template changes actual OpenCode behavior in a clean Java/Spring backend workflow.

The narrow question is:

- does OpenCode run and record `gradle test`;
- does OpenCode run and record `gradle build`;
- does OpenCode run and record `gradle bootRun`;
- does OpenCode run and record an HTTP happy path;
- does OpenCode run and record an HTTP failure path.

This is workflow evidence. It is not product-quality certification and not an enforcement gate.

## Fixture

The clean project requested a Club Event Registration API:

- create events;
- list events;
- create members;
- register a member for an event;
- fail when an event is full;
- fail duplicate member/event registration;
- allow a member to cancel only their own registration;
- return exception responses for missing event/member/registration cases.

The profile selected:

- team project;
- production service;
- small scale;
- REST API;
- database storage;
- `JdbcTemplate`;
- Flyway;
- domain-first package style;
- light Clean Architecture;
- strict boundaries.

The profile note also stated that domain models should not be records and should have state plus judgment methods.

## Generated Shape

The generated project kept the intended backend shape:

- Gradle only; no `pom.xml`;
- root application class at `com.example.ClubEventRegistrationApplication`;
- `global` package for shared exception handling;
- `clubevent.presentation`, `clubevent.application`, `clubevent.domain`, and `clubevent.infrastructure` packages;
- domain repository ports in `domain`;
- `JdbcTemplate` repository implementations in `infrastructure`;
- application services depending on domain repository ports;
- domain model classes with behavior:
  - `Event.canRegister`;
  - `Event.isFull`;
  - `Event.increaseRegisteredCount`;
  - `Event.decreaseRegisteredCount`;
  - `Registration.belongsToMember`;
  - `Registration.belongsToEvent`;
- no `Map`, `AtomicLong`, `nextId`, or `idCounter` ownership in production application services.

DTOs and global response objects used records, but domain models were class-based state holders. That matches the current backend Clean Code target.

The tightened implementation-report template did change the observed OpenCode behavior in this run:

- OpenCode ran and recorded `gradle test`;
- OpenCode ran and recorded `gradle build`;
- OpenCode ran and recorded `gradle bootRun`;
- OpenCode ran and recorded an HTTP happy path;
- OpenCode ran and recorded an HTTP failure path.

## Known Gaps

- `.persona/workflow/implementation-report.md` still kept `Status: template` even after OpenCode filled the report.
- Hidden `.persona` discovery stayed noisy: broad glob discovery did not reliably surface `.persona/workflow/plan.md`, although direct file reads worked.
- Broad Java globs can still see installed package fixture Java files under `node_modules/persona-harness`, even though direct evidence capture ignores those targets.
- The first build attempt selected Java 17, but the local machine only had Java 21. OpenCode repaired this by changing the build to Java 21.
- The first test run also needed `testRuntimeOnly 'org.junit.platform:junit-platform-launcher'`; OpenCode repaired this.
- Generated Java target-role injection was not exercised after source files existed.

## Decision

Keep the tightened live HTTP QA template.

Do not add an enforcement gate from this result. The useful next narrow improvement is report lifecycle clarity: a filled implementation report should not continue saying `Status: template`.

## Next

Recommended next loop:

Define and implement a narrow workflow report status lifecycle so generated reports can move from `template` to `filled` or `completed` without turning report completion into a build/test gate.
