# v0.3.0 Live HTTP QA Template Smoke

## Goal

Tighten the workflow report templates so implementation agents are prompted to run live HTTP manual QA for runnable Java/Spring apps, not only `gradle test` and `gradle build`.

This is workflow evidence guidance, not rule enforcement and not product-quality certification.

## Trigger

The accepted interactive profile implementation smoke generated a good Equipment Rental API, but OpenCode itself did not run live HTTP manual QA. It left the implementation report manual QA item unchecked and relied on MockMvc plus Gradle verification. A human follow-up then ran `gradle bootRun` and live HTTP requests.

## Changed Template

`ph plan` now writes `.persona/workflow/implementation-report.md` with explicit slots:

```text
- [ ] `gradle test`
- [ ] `gradle build`
- [ ] 실행 가능한 Spring Boot 앱이면 `gradle bootRun`

## Manual QA

- [ ] HTTP happy path를 curl 또는 동등한 HTTP client로 실행했다.
- [ ] HTTP failure path를 curl 또는 동등한 HTTP client로 실행했다.
- [ ] Manual QA가 불가능하면 사유와 stderr/핵심 로그를 기록한다.
```

The generated review report now asks reviewers to check:

```text
- [ ] 실행 가능한 Spring Boot 앱이면 `gradle bootRun` 결과를 확인했다.
- [ ] HTTP happy path / failure path manual QA evidence를 확인했다.
```

## Tests

`tests/persona-harness-plan.test.ts` now fixes the implementation and review report strings so future template drift is visible.

Targeted test:

```bash
npx vitest run tests/persona-harness-plan.test.ts tests/persona-harness-history.test.ts
```

Result: PASS.

## Finding

PASS.

The local `ph plan` surface now makes live HTTP QA explicit for runnable Spring Boot apps and gives the agent a fallback path when live QA cannot run.

## Limitations

- This does not force OpenCode to run HTTP QA.
- This does not fail builds/tests when the report is incomplete.
- This does not certify generated app quality.
- This smoke did not run a new OpenCode implementation after the template change.
