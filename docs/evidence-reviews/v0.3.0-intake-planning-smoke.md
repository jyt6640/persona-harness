# v0.3.0 Intake Planning Smoke

## Goal

Verify whether the new `ph intake` planning surface can guide OpenCode to read project profile answers and propose an architecture/technology plan before implementation.

This smoke does not certify generated app quality. It checks the planning surface only.

## Setup

- Persona Harness repo: `/Users/yongtae/Desktop/persona-harness`
- Clean smoke project: `/tmp/persona-v030-intake-planning-smoke`
- Model: `openai/gpt-5.4-mini-fast`
- Install mode: local path install from the current repo
- Commands:

```bash
npm install -D /Users/yongtae/Desktop/persona-harness
npx --no-install ph init
npx --no-install ph intake
```

`.persona/project-profile.jsonc` was filled with:

## OpenCode Prompt

```text
README.md와 .persona/project-profile.jsonc를 끝까지 읽어라.
코드는 만들거나 수정하지 말고, 구현 전에 필요한 architecture/technology plan만 한국어로 제안해라.
반드시 profile의 project-context, project-scale, storage, persistence-technology,
migration-style, package-style, dto-strictness, philosophy-overlay 답변을 반영하고,
마지막에 'IMPLEMENTATION_NOT_STARTED'라고 써라.
```

## Result

PASS.

OpenCode:

- read `README.md`;
- read `.persona/project-profile.jsonc`;
- reflected all eight profile answers in the plan;
- proposed Java/Spring Boot/Gradle, in-memory storage, domain-first packages, strict DTO boundaries, and repository/storage separation;
- ended with `IMPLEMENTATION_NOT_STARTED`;
- did not generate `src/main/java` or implementation files.

Runtime evidence was created under:

```text
/tmp/persona-v030-intake-planning-smoke/.persona/evidence/phase0
```

Observed evidence count: 4 JSON files.

## Finding

`ph intake` is a viable v0.3.0 planning surface. It can make project-shaping decisions visible before generation without turning them into rule enforcement or product-quality certification.

## Caveats

- `ph intake` is still a draft generator, not an interactive interview.
- The profile is not automatically injected as a dedicated rule block; the prompt explicitly told OpenCode to read it.
- The smoke verifies planning behavior only, not code generation quality after using the plan.
- OpenCode installed local `.opencode` runtime dependencies inside the smoke project during execution. That is OpenCode behavior, not Persona Harness template copying.

## Decision

Keep `ph intake` as the next active v0.3.0 surface.

Next useful loop:

1. Run a planned implementation smoke where OpenCode reads `README.md` and `.persona/project-profile.jsonc`, proposes a plan, then implements only after that plan.
2. Compare the generated shape against the previous README-only runs.
3. Decide whether project profile answers should be automatically referenced by Persona Harness injection or remain prompt/read-driven for now.
