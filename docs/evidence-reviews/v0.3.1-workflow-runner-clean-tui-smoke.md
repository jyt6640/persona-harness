## Comparison To Guard Smoke

Previous guard smoke:

- 3/3 entered the implement guard.
- 3/3 ran `npx ph plan --implement`.
- 3/3 filled implementation and review reports.
- 1/3 reached final guard PASS naturally.

Runner smoke:

- 2/2 ran `npx ph workflow start implement`.
- 2/2 ran `npx ph plan --implement`.
- 2/2 filled implementation and review reports.
- 1/2 reached final workflow PASS.
- 2/2 produced buildable Java/Spring Gradle apps with strong backend shape by manual inspection.

The runner is easier for the model to follow than separate low-level guard commands, but it does not fully solve command discipline.

## Decision

Treat `ph workflow start implement` / `ph workflow finish implement` as a useful improvement, not a closed workflow discipline solution.

The next improvement should stay narrow:

1. Detect and report direct `.persona/rules` reads as workflow noise, because selected rules should be delivered through injection summaries and the accepted plan.
2. Decide whether environment probes such as `java -version` and `gradle -v` should be allowed outside `npx ph bearshell`, or whether all shell commands should be routed through `npx ph bearshell`.

Do not add more backend shape rules from this smoke. Backend shape was already strong in both generated apps.

## Limitations

- Only two clean OpenCode runs.
- OpenCode model behavior can vary.
- Manual backend shape review is evidence, not certification.
- `workflow check` and `workflow finish` are AI-facing workflow rails, not generated app quality gates.
- The smoke used local path install, not public npm install.

## Next

Narrow follow-up candidate:

`ph workflow finish implement` should distinguish command-discipline categories:

- blocking: final verification/build/test/bootRun performed only through raw shell;
- warning: early environment probes used raw shell but final verification was repeated through `npx ph bearshell`;
- warning: direct `.persona/rules` read observed in clean implementation context.

This keeps the focus on AI workflow discipline instead of expanding product-quality enforcement.
