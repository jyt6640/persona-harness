# v0.3.0 Policy Overlay Clean Workflow Smoke

## Goal

Verify the clean backend workflow:

```text
ph init -> ph intake --interactive -> ph policy init -> ph plan -> OpenCode plan-only
```

The check focuses on whether OpenCode reads the policy overlay as planning context, not as an enforcement rule or product-quality guarantee.

## Setup

- Project path: `/tmp/persona-policy-clean-smoke-qw10F6`
- Install mode: local path install from `/Users/yongtae/Desktop/persona-harness`
- Model: `openai/gpt-5.4-mini-fast`
- Prompt mode: plan-only
- Fixture: Team Project Tracker API

README requirements:

- register members;
- create projects;
- assign members to projects;
- list projects with assigned member count;
- mark a project as completed;
- completed projects cannot receive new members;
- missing member/project returns an appropriate error response;
- Gradle backend only, no frontend or infrastructure deployment.

## Workflow Executed

1. Created a clean temporary project.
2. Installed Persona Harness from the local workspace.
3. Ran `npx ph init`.
4. Ran `npx ph intake --interactive` through a real PTY session.
5. Ran `npx ph policy init`.
6. Filled backend company and personal policy files.
7. Ran `npx ph plan`.
8. Ran OpenCode with a plan-only prompt that explicitly required reading:
   - `README.md`
   - `.persona/project-profile.jsonc`
   - `.persona/policies/overlay.jsonc`
   - `.persona/policies/company/backend.md`
   - `.persona/policies/personal/backend.md`
   - `.persona/workflow/plan.md`

## Policy Inputs

Company policy:

- Prefer domain-first backend packages when the README does not require a different structure.
- Keep repository ports in the domain and concrete persistence adapters in infrastructure.

Personal philosophy:

- Domain models should be classes with behavior, not passive records.
- Application Services should orchestrate use cases and avoid owning storage state or id sequences.

## OpenCode Result

OpenCode completed with exit status `0`.

Observed behavior:

- OpenCode read README, project profile, policy overlay, company policy, personal policy, and workflow plan.
- OpenCode patched only `.persona/workflow/plan.md`.
- No `src/` directory was created.
- No `build.gradle` was created.
- No implementation was started.
- OpenCode stated that policy overlay was planning context only and not enforcement.

## Troubleshooting

The first attempt used `script` to pipe all `ph intake --interactive` answers at once. That produced EOF at the first prompt and no `.persona/project-profile.jsonc`.

The successful run used a real PTY session and answered one prompt at a time. This matters because `ph intake --interactive` intentionally requires TTY input.

## Decision

PASS for plan-only smoke.

Evidence supports:

- `ph init`, interactive intake, policy init, and plan draft work together in a clean project.
- policy overlay appears in `ph plan` and OpenCode's completed architecture plan.
- OpenCode treated the overlay as planning context, not enforcement.
- README/current prompt requirements stayed above policy overlay.

## Limits

- This did not implement the generated backend app.
- This did not verify whether policy overlay changes implementation output after plan acceptance.
- This is not product-quality certification.
- This is not a rule enforcement gate.
- This does not activate frontend/infra/global policy loading.
