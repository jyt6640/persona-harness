## Fixture

The clean project requested a Study Group Session API:

- create study groups;
- list study groups;
- create members;
- create study sessions;
- reserve a session;
- reject full sessions;
- reject duplicate member/session reservations;
- allow a member to cancel only their own reservation;
- return appropriate exception responses for missing group/member/reservation cases.

Technical direction:

- Java/Spring Boot;
- Gradle only;
- REST API;
- domain-first package structure;
- `presentation/application/domain/infrastructure` boundaries;
- domain repository ports;
- `JdbcTemplate` infrastructure repositories;
- class-based domain models with state and behavior;
- no Service-owned storage/id sequence.

## Important Caveat

The follow-up review exposed a rule-scope concern:

- `backend/step1-api-contract.md` appeared on arbitrary clean-project Controller/DTO targets.
- The generated API used REST-appropriate `201 Created` and `204 No Content` responses.
- The injected step1 contract expects `200 OK`, so OpenCode recorded low-severity review findings about a contract mismatch.

This is useful evidence that role-specific injection works, but it also suggests the old step fixture API contract should not automatically apply to every v0.3.0 clean backend project.

## Known Limits

- This was one clean generated project.
- The target-role follow-up was explicitly prompted; this does not prove OpenCode will naturally read every role file without instruction.
- `ph intake --interactive` was not used because this non-TTY smoke could not answer interactive prompts.
- The generated app is not certified as product-quality complete.

## Decision

Keep generated Java target-role read follow-up as a verified v0.3.0 workflow capability.

Do not treat this as evidence for more product-quality enforcement. The next narrow decision should be whether step fixture API contract rules need stricter scoping before they are used in arbitrary clean-project workflows.
