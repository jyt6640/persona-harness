# v0.3.0-alpha.2 TUI Short Implementation Review

## Goal

Verify whether a short OpenCode instruction can drive the current Persona Harness workflow:

```text
플랜 보고 구현해줘
```

The review checks whether the model:

- runs `npx ph plan --implement` first
- reads README/plan in more than one chunk instead of stopping at the first read
- records remaining work in `Continuation` if it stops early
- includes late README requirements such as `theme` and `waiting`
- fills `.persona/workflow/implementation-report.md`

## Result

Finding: `MIXED_POSITIVE`

The plan-aware workflow mostly worked. The model ran the accepted plan gate, implemented `reservations`, `times`, `themes`, and `waiting`, ran Gradle verification, ran HTTP smoke checks, filled the implementation report, and completed without leaving remaining implementation scope.

The exact read behavior was weaker than the prompt requested: the model did not use `ph bearshell --shell 'sed -n ...'` for README chunk reads. It used OpenCode `Read README.md [offset=1, limit=250]` followed by `Read README.md [offset=251, limit=80]`. This still proves it did not stop at the first README chunk, but it does not prove the desired `bearshell` line-range habit.

The implementation report also overclaimed read coverage by checking the `ph bearshell sed` item even though the log shows native OpenCode Read offsets. That makes the report useful but not fully reliable as evidence.

## Cold Analysis

This is not a failure of the plan-aware workflow. The core flow worked:

```text
short prompt -> plan gate -> late README read -> implementation -> verification -> report filled
```

But it is also not a clean alpha3 release signal yet. The weak point is evidence honesty. The model checked a report item claiming `bearshell sed` read coverage without actually using that command. A report checkbox is too easy to satisfy by assertion.

The next improvement should not be more backend rules. It should narrow the report evidence shape:

- record actual read method
- record actual read ranges
- allow either OpenCode Read offsets or `ph bearshell sed` ranges
- avoid a checkbox that implies a specific command was run when it was not

## Decision

Decision: `continue-alpha-hardening`

Do not move straight to `0.3.0-alpha.3` solely from this run.

Recommended next change:

```text
Make Read Coverage evidence explicit instead of checkbox-only.
```

After that, run the same short TUI prompt again. If it again covers late requirements and produces honest read coverage evidence, `0.3.0-alpha.3` is a reasonable candidate.
