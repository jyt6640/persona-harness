## Policy Overlay Influence

Positive signals:

- Domain-first package shape was used.
- Repository ports were placed in `domain/repository`.
- Concrete JDBC adapters were placed in `infrastructure/persistence`.
- `Project` is a class with behavior, not a passive record.
- `Project.complete()` and `Project.assignMember()` own completion-state decisions.
- Application Service orchestrates use cases and depends on domain repository ports.
- No Service-owned `Map`, `List`, `AtomicLong`, `nextId`, or `idCounter` storage/id-sequence field was observed.


## Known Gap

OpenCode reintroduced the `bootJar` workaround:

```groovy
bootJar {
    enabled = false
}

jar {
    enabled = true
}
```

This makes `gradle build` pass with `:bootJar SKIPPED`, but it weakens the executable Spring Boot artifact expectation captured in the Gradle/Spring build-line guidance. `gradle bootRun` and HTTP smoke still passed, so runtime behavior was verified, but build-line guidance regressed.

This should not be fixed by broad policy/enforcement. The next loop should narrowly address why OpenCode still disables `bootJar` when it hits the local Gradle 9.4 / Spring Boot 3.3.2 `CopyProcessingSpec.getDirMode()` mismatch. The better direction is to make the agent choose a compatible Spring Boot/Gradle line instead of disabling `bootJar`.

## Decision

PASS for policy-overlay accepted implementation behavior.

PARTIAL for packaging/build-line behavior because `bootJar` was disabled.
