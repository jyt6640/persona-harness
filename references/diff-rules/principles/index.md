# Principles

이 문서는 프로젝트의 핵심 코드 작성 원칙을 정리한다.

세부 규칙은 docs/principles 하위 문서를 따른다.

---

## 핵심 방향

- 객체는 자기 상태와 규칙을 스스로 관리한다.
- 흐름과 판단을 분리한다.
- 명시성을 재사용보다 우선한다.
- 이름은 구현이 아니라 의도를 표현해야 한다.
- 구조는 먼저 지키되, 추상화는 필요해질 때 도입한다.
- 작은 결정의 일관성이 전체 코드 품질을 만든다.

---

## OOP

객체의 책임과 협력 기준을 정의한다.

- 객체는 자기 상태를 스스로 검증한다.
- Getter보다 행위를 우선한다.
- 책임은 데이터를 가진 객체에 둔다.
- 가능한 불변성을 유지한다.

→ [oop.md](oop.md)

---

## Method Design

메서드 단위 설계 기준을 정의한다.

- 한 메서드는 하나의 의도만 표현한다.
- 흐름과 판단을 분리한다.
- else보다 early return/throw를 우선한다.
- boolean 파라미터로 정책 차이를 숨기지 않는다.

→ [method-design.md](method-design.md)

---

## Naming

네이밍 기준을 정의한다.

- 이름은 도메인 모델링의 결과다.
- 역할이 드러나는 이름을 우선한다.
- Manager / Helper / Util 같은 모호한 이름을 지양한다.

→ [naming.md](naming.md)

---

## Exceptions

예외 처리 기준을 정의한다.

- 예외를 던지는 위치는 검증 책임 위치와 일치해야 한다.
- 도메인별 ErrorCode를 분리한다.
- 모든 커스텀 예외는 unchecked로 처리한다.

→ [exceptions.md](exceptions.md)

---

## Testing

테스트 작성 기준을 정의한다.

- 테스트는 구현보다 행위를 검증한다.
- Fake와 Mock는 목적에 따라 선택한다.
- 테스트 가능한 구조를 우선한다.

→ [testing.md](testing.md)

## 판단 기준

코드 작성 기준이 헷갈리면 아래 질문으로 판단한다.

- 이름만으로 의도를 이해 가능한가?
- 객체가 자기 책임을 자연스럽게 가지고 있는가?
- 흐름과 판단이 섞여 있지 않은가?
- 불필요한 추상화가 존재하지 않는가?
- 구조보다 구현 디테일이 먼저 보이지 않는가?

하나라도 NO라면 구조를 다시 검토한다.
