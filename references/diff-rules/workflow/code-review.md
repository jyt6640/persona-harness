# Code Review

이 문서는 프로젝트의 코드 리뷰 기준을 정의한다.

코드 리뷰는 단순 오류 탐지가 아니라,
설계와 의도를 함께 검토하는 과정으로 본다.

---

## 핵심 방향

- 코드보다 의도를 먼저 이해한다.
- 정답 강요보다 판단 근거를 공유한다.
- 작은 피드백도 구조 신호로 본다.
- 코드 리뷰는 설계 학습 과정이다.
- 현재 프로젝트 기준을 우선한다.

---

## 리뷰 기준

아래 기준으로 코드를 검토한다.

- 책임이 자연스럽게 배치되어 있는가?
- 이름만으로 의도를 이해 가능한가?
- 흐름과 판단이 분리되어 있는가?
- 레이어 책임이 섞여 있지 않은가?
- 불필요한 추상화가 존재하지 않는가?

---

## 네이밍 리뷰

이름은 가장 먼저 검토한다.

아래 상황은 재검토 신호다.

- 이름이 구현을 설명한다.
- Manager / Helper / Util 같은 모호한 이름 사용
- 메서드 이름에 and 포함
- 역할보다 기술 용어가 먼저 보임

좋은 예시:

    validateOwner()
    cancel()
    OrderPolicy

지양하는 예시:

    process()
    handle()
    OrderManager

---

## 구조 리뷰

아래 상황은 구조 재검토 신호로 본다.

- Service에 정책 판단 집중
- getter를 꺼내 외부에서 판단
- Repository가 비즈니스 로직 수행
- Domain이 단순 DTO처럼 동작
- util/helper 클래스 증가

---

## 테스트 리뷰

테스트도 프로덕션 코드처럼 리뷰한다.

### 기준

- 테스트 이름만으로 의도를 이해 가능한가?
- 구현 세부사항에 과하게 의존하지 않는가?
- 중요한 비즈니스 규칙이 검증되는가?
- Fake / Mock 선택 이유가 명확한가?

---

## 피드백 방식

리뷰는 수정 강요보다 판단 근거를 우선한다.

좋은 예시:

    "이 책임은 Domain 쪽이 더 자연스러워 보입니다."
    "조건 분리가 Policy 신호처럼 보입니다."

지양하는 예시:

    "무조건 이렇게 해야 합니다."

---

## 피드백 처리 기준

리뷰 의견은 바로 수용하거나 거부하지 않는다.

### 흐름

1. 일단 이해한다.
2. 직접 적용해본다.
3. 현재 구조와 비교한다.
4. 더 자연스러운 방향을 선택한다.

---

## 보류 가능한 영역

아래 상황은 즉시 결정하지 않고 보류할 수 있다.

- 프로젝트 규모에 따라 답이 달라진다.
- 아직 사용 사례가 충분하지 않다.
- 구조보다 경험 축적이 우선이다.

보류는 거절이 아니라,
판단 시점을 뒤로 미루는 선택으로 본다.

---

## 지양하는 리뷰

- 취향만으로 수정 요구
- 근거 없는 일반론 적용
- 현재 프로젝트 맥락 무시
- 과도한 미래 확장 설계 강요
- 구현 디테일만 보고 의도 무시

---

## 판단 기준

리뷰가 헷갈리면 아래 질문으로 판단한다.

- 현재 프로젝트 구조와 일관되는가?
- 책임이 더 자연스러워지는가?
- 코드 의도가 더 명확해지는가?
- 유지보수 시 이해가 쉬워지는가?

YES라면 좋은 리뷰 가능성이 높다.
