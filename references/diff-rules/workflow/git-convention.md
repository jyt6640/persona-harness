# Git Convention

이 문서는 프로젝트의 Git 작업 기준을 정의한다.

Git은 단순 버전 관리 도구가 아니라,
작업 흐름과 변경 의도를 기록하는 수단으로 본다.

---

## 핵심 방향

- 작은 단위로 자주 커밋한다.
- 커밋 하나는 하나의 의도를 가진다.
- 변경 이유를 추적 가능해야 한다.
- 기능 단위로 PR을 구성한다.
- 리뷰 가능한 크기를 유지한다.

---

## 커밋 단위

커밋은 public behavior 단위로 나눈다.

기본 순서:

1. 실패하는 테스트를 추가한다.
2. 실패를 확인한다.
3. 테스트 커밋을 만든다.
4. 최소 구현으로 테스트를 통과시킨다.
5. 구현 커밋을 만든다.
6. 리팩터링이 필요하면 별도 커밋으로 분리한다.

큰 기능명 기준으로 한 번에 커밋하지 않는다.

지양:

    feat(member): 회원가입 구현

권장:

    test(member): 이메일 형식 검증 테스트 추가
    feat(member): 이메일 형식 검증 구현
    test(member): 이메일 중복 검증 테스트 추가
    feat(member): 이메일 중복 검증 구현

---

## Commit Message

커밋 제목은 Conventional Commit 형식을 따른다.

형식:

    type(scope): summary

규칙:

- type과 scope는 영어로 작성한다.
- summary는 한국어로 작성한다.
- 본문은 항상 한국어로 작성한다.
- 본문에는 변경 이유, 테스트 범위, 다음 작업과의 경계를 적는다.
- 테스트 커밋과 구현 커밋을 분리한다.

예시:

    test(member): 중복 이메일 검증 테스트 추가

    회원가입 시 이메일 중복 검증 책임을 MemberValidator에 고정한다.

    - Repository 조회가 필요한 검증을 Application Validator에서 수행한다.
    - Service 테스트가 중복 이메일 정책 검증을 대신하지 않도록 단위 테스트를 추가한다.

---

## 커밋 타입

### feat

새 기능 추가

### fix

버그 수정

### refactor

동작 변경 없는 구조 개선

### test

테스트 추가 및 수정

### docs

문서 수정

### chore

설정, 빌드, 기타 작업

---

## 브랜치 전략

브랜치는 작업 목적 단위로 생성한다.

좋은 예시:

    feat/order-cancel
    refactor/validator-separation
    fix/duplicate-validation

지양하는 예시:

    test
    temp
    final-final-real

---

## Pull Request 기준

PR은 기능 단위로 올린다.

### 좋은 기준

- 하나의 사용자 시나리오
- 하나의 책임 변경
- 리뷰 가능한 크기

### 지양하는 기준

- 여러 기능 혼합
- 대규모 구조 변경 한 번에 포함
- 의도를 설명하기 어려운 변경

---

## PR 설명

PR에는 아래 내용을 포함한다.

- 무엇을 변경했는가
- 왜 변경했는가
- 어떤 기준으로 설계했는가
- 리뷰 시 중점적으로 볼 부분은 무엇인가

---

## 보강 / 리팩터링 커밋

코드 리뷰 보강, 테스트 보강, 리팩터링도 커밋 단위를 지킨다.

### 코드 리뷰 보강

코드 리뷰에서 발견한 문제는 하나의 책임 또는 하나의 public behavior 단위로 수정한다.

좋은 예시:

    test(reservation): 예약 변경 중복 검증 테스트 보강
    feat(reservation): 예약 변경 중복 검증 처리
    refactor(reservation): 예약 변경 검증 책임 분리

지양하는 예시:

    fix: 리뷰 반영
    refactor: 전체 정리

### 테스트 보강

테스트 보강 커밋은 하나의 누락 행위 또는 하나의 누락 책임만 포함한다.

좋은 예시:

    test(member): 회원 생성 공백 입력 테스트 보강
    test(reservation): 예약 취소 소유자 검증 테스트 보강

지양하는 예시:

    test: 테스트 보강

### 리팩터링

리팩터링 커밋은 행위 변경 없이 하나의 구조 개선만 포함한다.

좋은 예시:

    refactor(test): 예약 서비스 fake 분리
    refactor(member): 회원 생성 정적 팩터리 분리

지양하는 예시:

    refactor: 테스트랑 구조 전체 수정

---

## 테스트 커밋

테스트 추가는 가능한 명확히 드러낸다.

테스트 커밋은 기대 행위나 실패 케이스를 먼저 표현한다.
이후 구현 커밋은 그 테스트를 통과시키는 변경만 포함한다.

좋은 예시:

    test(order): 주문 취소 권한 검증 테스트 추가
    feat(order): 주문 취소 권한 검증 구현

---

## 지양하는 방식

- 의미 없는 대규모 커밋
- 변경 이유 없는 force push 반복
- 테스트 없이 기능만 추가
- "최종", "진짜최종" 같은 브랜치 이름
- 동작 변경과 리팩토링을 한 커밋에 혼합
- 코드 리뷰 보강을 하나의 큰 커밋으로 처리
- 테스트 보강과 구현 변경을 한 커밋에 혼합
- 리팩터링과 동작 변경을 한 커밋에 혼합

---

## 판단 기준

커밋과 PR 구성이 헷갈리면 아래 질문으로 판단한다.

- 변경 이유를 한 문장으로 설명 가능한가?
- 리뷰어가 흐름을 따라가기 쉬운가?
- 되돌려야 할 때 영향 범위를 예측 가능한가?
- 기능과 구조 변경이 자연스럽게 분리되어 있는가?

YES라면 좋은 Git 단위 가능성이 높다.
