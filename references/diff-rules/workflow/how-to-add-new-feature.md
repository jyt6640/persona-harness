# How To Add New Feature

이 문서는 새로운 기능을 추가할 때의 기본 작업 흐름을 정의한다.

기능 구현은 단순 코드 추가가 아니라,
도메인 규칙과 구조를 함께 설계하는 과정으로 본다.

---

## 핵심 방향

- 기능은 작은 단위로 구현한다.
- 안쪽 도메인부터 바깥 방향으로 구현한다.
- 흐름과 정책을 분리한다.
- 테스트 가능한 상태를 유지한다.
- 현재 구조와 일관성을 우선한다.

---

## 기본 구현 순서

기능은 아래 순서로 구현한다.

1. 요구사항 이해
2. Domain 설계
3. Domain Test 작성
4. Service 흐름 작성
5. Repository 연결
6. Controller 연결
7. Acceptance Test 작성
8. 리팩토링

---

## 1. 요구사항 이해

먼저 아래 내용을 정리한다.

- 핵심 비즈니스 규칙은 무엇인가
- 상태 변경 조건은 무엇인가
- 실패 케이스는 무엇인가
- 어떤 객체가 책임져야 하는가

구현 전에:
- 흐름
- 정책
- 책임 위치

를 먼저 고민한다.

---

## 2. Domain 설계

핵심 규칙은 Domain부터 설계한다.

우선 고민할 것:

- 어떤 상태를 가지는가
- 어떤 행위를 수행하는가
- 어떤 검증이 필요한가
- 어떤 규칙을 스스로 알아야 하는가

---

## 3. Domain Test 작성

중요한 비즈니스 규칙은 먼저 테스트로 표현한다.

예시:

- 취소 가능 여부
- 중복 불가 규칙
- 상태 변경 조건

테스트는:
- 구현
보다
- 기대 행위

를 먼저 표현한다.

---

## 4. Service 흐름 작성

Service는:
- 유스케이스 흐름
- 객체 협력
- 트랜잭션 경계

를 조율한다.

정책 판단은:
- Domain
- Policy
- Validator

에 위임한다.

---

## 5. Repository 연결

Repository는 저장 기술 구현만 담당한다.

주의:
- Repository에서 비즈니스 로직 처리 금지
- 저장소 접근과 정책 판단 혼합 금지

---

## 6. Controller 연결

Controller는:
- HTTP 요청/응답
- DTO 변환
- 상태 코드 처리

만 담당한다.

비즈니스 로직을 직접 처리하지 않는다.

---

## 7. Acceptance Test 작성

외부 사용자 관점에서 흐름을 검증한다.

예시:

- 주문 생성 성공
- 예약 취소 실패
- 권한 없는 접근 차단

---

## 8. 리팩토링

기능 구현 후 아래 신호를 확인한다.

- 이름이 어색한가
- 책임이 한 곳에 몰렸는가
- 조건문이 증가했는가
- util/helper가 생기기 시작하는가
- Service가 정책 판단을 하는가

---

## 구조 우선순위

기능 추가 시 아래 우선순위를 유지한다.

1. 명확한 이름
2. 책임 분리
3. 테스트 가능성
4. 흐름 가독성
5. 재사용성

---

## 지양하는 방식

- Controller부터 구현 시작
- Service에 모든 로직 집중
- 테스트 없이 핵심 규칙 작성
- util/helper로 빠른 공통화
- 미래 확장을 위한 선제 추상화

---

## 마지막 원칙

기능 구현 중 기존 문서 기준과 충돌하면:

- 현재 구조가 맞는지
- 문서가 오래된 것인지

함께 다시 검토한다.
