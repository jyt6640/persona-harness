# How To Review Legacy Code

이 문서는 기존 코드를 분석하고 수정할 때의 기준을 정의한다.

레거시 코드는 단순히 오래된 코드가 아니라,
의도를 이해하기 어려운 상태의 코드로 본다.

---

## 핵심 방향

- 먼저 이해하고 나중에 수정한다.
- 현재 구조의 의도를 먼저 파악한다.
- 한 번에 전체를 바꾸지 않는다.
- 작은 단위로 안전하게 개선한다.
- 테스트 가능한 상태를 먼저 만든다.

---

## 우선 확인할 것

기존 코드를 수정하기 전에 아래를 먼저 확인한다.

- 핵심 흐름은 무엇인가
- 상태 변경 지점은 어디인가
- 검증 책임은 어디 있는가
- 어떤 객체가 정책을 가지는가
- 어떤 테스트가 존재하는가

---

## 바로 리팩토링하지 않는다

처음부터 구조를 갈아엎지 않는다.

먼저:
- 흐름 이해
- 책임 파악
- 변경 영향 분석

을 우선한다.

---

## 구조 신호 확인

아래 상황은 구조 개선 신호로 본다.

- Service가 모든 판단 수행
- getter 기반 조건문 반복
- util/helper 증가
- 조건문 depth 증가
- 이름만으로 역할 이해 불가
- Repository가 정책까지 처리

---

## 작은 단위 수정

좋은 예시:

- 메서드 이름 수정
- 책임 하나 이동
- Validator 분리
- Policy 분리

지양하는 예시:

- 전체 구조 재작성
- 패턴 전체 교체
- 대규모 리아키텍처

---

## 테스트 우선 확보

핵심 흐름 수정 전:
- 기존 동작을 보호하는 테스트
를 먼저 만든다.

특히:
- 상태 변경
- 정책 판단
- 실패 케이스

를 우선 보호한다.

---

## 흐름과 판단 분리

아래 상황이면 분리를 검토한다.

### Service

- 흐름 orchestration만 담당

### Domain / Policy

- 정책 판단
- 상태 검증

### Validator

- 저장소 조회 기반 검증

---

## util/helper 주의

레거시 코드에서는 아래 구조가 자주 등장한다.

    CommonUtil
    ValidationHelper
    OrderManager

이 경우:
- 실제 책임
- 도메인 의미
- 호출 맥락

을 먼저 분석한다.

바로 삭제하지 않는다.

---

## 이름 먼저 수정

이름이 이상하면:
- 구조 문제
- 책임 혼합
- 의미 누락

신호일 가능성이 높다.

리팩토링은 이름부터 시작한다.

---

## 보류 가능한 영역

아래 상황은 즉시 수정하지 않을 수 있다.

- 현재 기능과 무관하다.
- 변경 영향 범위가 너무 크다.
- 테스트 보호가 부족하다.
- 실제 문제 신호가 아직 약하다.

---

## 마지막 원칙

레거시 코드를 수정할 때는:

"완벽한 구조"
보다
"안전한 개선"

을 우선한다.
