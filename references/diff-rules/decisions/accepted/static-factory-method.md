# Static Factory Method

## 상태

accepted

---

## 문제 상황

도메인 객체 생성을 아래 방식 중 어떤 기준으로 할지 결정이 필요했다.

### public 생성자 사용

    new Order(...)

### 정적 팩토리 메서드 사용

    Order.create(...)
    Order.restore(...)

---

## 선택한 방향

도메인 객체 생성은 정적 팩토리 메서드를 우선 사용한다.

생성자는 외부에 직접 노출하지 않고,
생성 의도에 맞는 이름을 가진 메서드로 진입한다.

---

## 현재 구조

### 신규 생성

비즈니스 검증이 필요한 생성이다.

    Order.create(...)

---

### 복원

저장소에서 조회한 데이터를 도메인 객체로 되돌리는 생성이다.

    Order.restore(...)

또는:

    Order.from(...)

---

## 선택 이유

### 생성 의도 표현

`new`는 객체 생성만 표현한다.

반면 정적 팩토리 메서드는:

- 신규 생성
- 복원
- 변환
- 테스트용 생성

같은 의도를 이름으로 드러낼 수 있다.

---

### 생성 경로 분리

신규 생성과 복원은 같은 객체를 만들더라도 의미가 다르다.

- 신규 생성: 비즈니스 검증 필요
- 복원: 이미 저장된 데이터 복원

따라서 생성 경로를 분리한다.

---

### 유효한 객체 생성 보장

객체가 생성되는 순간부터
가능한 유효한 상태를 유지하도록 한다.

생성 규칙을 객체 내부에 모으면:
- 검증 누락 감소
- 생성 흐름 명확화
- 불변성 유지

에 도움이 된다.

---

## 좋은 예시

    Order.create(customerId, items)
    Order.restore(id, customerId, status)

---

## 지양하는 예시

    new Order(...)
    order.setId(id)

---

## 트레이드오프

### 생성 메서드 증가 가능

생성 맥락이 늘어나면
정적 팩토리 메서드도 늘어날 수 있다.

---

### 단순 객체에서는 과할 수 있음

값 전달만 하는 단순 DTO는
record 생성자 사용이 더 자연스럽다.

---

## 현재 판단

현재 프로젝트는:

- 생성 의도 명확성
- 유효 상태 보장
- 불변성 유지

를 더 중요하게 판단한다.

따라서 도메인 객체는 정적 팩토리 메서드 기반 생성을 우선한다.

---

## 재검토 신호

아래 상황이면 구조를 다시 검토한다.

- 생성 메서드가 과도하게 증가한다.
- create / restore 의미가 흐려진다.
- 단순 데이터 객체에도 불필요하게 적용된다.
- 생성 로직보다 이름 관리 비용이 커진다.
