# Domain First Package Structure

## 상태

accepted

---

## 문제 상황

패키지 구조를 아래 두 방식 중 어떤 기준으로 구성할지 결정이 필요했다.

### 레이어 우선 구조

    controller/
    service/
    repository/

### 도메인 우선 구조

    order/
    user/
    payment/

---

## 선택한 방향

도메인 우선 패키지 구조를 사용한다.

예시:

    project
    ├── order
    │   ├── presentation
    │   ├── application
    │   ├── domain
    │   └── infrastructure

---

## 선택 이유

### 도메인 응집도 증가

하나의 도메인과 관련된 코드가 같은 위치에 모인다.

- 흐름 추적이 쉽다.
- 변경 영향 범위를 파악하기 쉽다.
- 도메인 단위로 사고 가능하다.

---

### 레이어 비대화 방지

레이어 우선 구조에서는 아래 문제가 발생하기 쉽다.

    service/
    ├── OrderService
    ├── UserService
    ├── PaymentService
    └── ...

도메인이 증가할수록 레이어 패키지가 과도하게 커진다.

---

### 기능 단위 개발 흐름과 일치

프로젝트는 기능 슬라이스 단위 개발을 지향한다.

즉:

    Domain
    → Service
    → Repository
    → Controller

를 하나의 흐름으로 구현한다.

도메인 우선 구조가 이 흐름과 더 자연스럽게 연결된다.

---

## 트레이드오프

### 공통 레이어 탐색 비용 증가 가능

특정 레이어 전체를 한 번에 보기 어려울 수 있다.

예시:
- 모든 Controller 찾기
- 모든 Repository 구현체 찾기

---

### 작은 프로젝트에서는 과할 수 있음

아주 작은 규모에서는 레이어 우선 구조가 더 단순할 수 있다.

---

## 현재 판단

현재 프로젝트 규모와 작업 방식에서는
도메인 우선 구조가 더 자연스럽다고 판단한다.

---

## 재검토 신호

아래 상황이면 구조를 다시 검토한다.

- 도메인 간 결합이 과도하게 증가한다.
- 특정 레이어 공통화 요구가 커진다.
- 모듈 단위 분리가 필요해진다.
- 패키지 탐색 비용이 커진다.
