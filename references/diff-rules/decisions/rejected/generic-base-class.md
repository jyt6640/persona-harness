# Generic Base Class

## 상태

rejected

---

## 문제 상황

중복 제거를 위해
공통 Base 클래스를 적극적으로 도입할지 고민했다.

예시:

    BaseService
    BaseController
    BaseEntity
    AbstractRepository

같은 구조다.

---

## 검토한 방향

### Generic Base Class 구조

장점:
- 중복 제거 가능
- 공통 로직 재사용 쉬움
- 빠른 확장 가능

단점:
- 상속 관계 복잡화
- 책임 흐려짐
- 의도 가독성 감소

---

## 선택한 방향

공통 부모 클래스보다
명시적인 역할 분리를 우선한다.

즉:
- 먼저 독립적으로 작성한다.
- 실제 반복 패턴이 충분히 드러난 뒤 공통화를 검토한다.
- 상속보다 조합을 우선 고려한다.

---

## 선택 이유

### 너무 이른 공통화 방지

초기 Base 클래스는:
- 미래 확장 예측
- 아직 없는 공통 규칙
- 불확실한 사용 패턴

을 기반으로 만들어지는 경우가 많다.

그 결과:
- 불필요한 추상 계층
- 의미 없는 override
- 구조 복잡도 증가

가 발생하기 쉽다.

---

### 책임 흐려짐

예시:

    BaseService

이 이름만으로는:
- 어떤 책임을 가지는지
- 어떤 규칙을 강제하는지
- 왜 상속받아야 하는지

명확하지 않다.

---

### 상속 결합 증가

Base 클래스 수정이
여러 객체에 동시에 영향을 줄 수 있다.

그 결과:
- 변경 영향 예측 어려움
- 테스트 범위 증가
- 숨겨진 의존성 증가

문제가 발생할 수 있다.

---

## 예시

좋은 예시:

    OrderValidator
    PaymentPolicy
    PriceCalculator

지양하는 예시:

    AbstractManager
    BaseService
    BaseController

---

## 조합 우선

공통 기능이 필요하면
상속보다 조합을 우선 고려한다.

좋은 예시:

    validator.validate()

지양하는 예시:

    extends BaseValidator

---

## 트레이드오프

### 중복 코드 증가 가능

초기에는 유사 구조가 반복될 수 있다.

---

### 공통 수정 비용 가능

동일 정책 변경 시 여러 위치를 수정해야 할 수 있다.

---

## 현재 판단

현재 프로젝트는:

- 의도 가독성
- 책임 명확성
- 구조 단순성

을 더 중요하게 판단한다.

따라서 Generic Base Class 구조는 채택하지 않는다.

---

## 재검토 신호

아래 상황이면 구조를 다시 검토한다.

- 같은 구조가 반복적으로 등장한다.
- 공통 수정 비용이 증가한다.
- 상속보다 조합 비용이 더 커진다.
- 공통 정책 관리가 어려워진다.
