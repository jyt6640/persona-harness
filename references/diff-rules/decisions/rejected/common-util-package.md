# Common Util Package

## 상태

rejected

---

## 문제 상황

공통 로직을 아래와 같은 패키지에 모을지 고민했다.

예시:

    common/
    util/
    shared/
    helper/

이 구조는:
- 빠르게 공통화 가능
- 중복 제거 쉬움
- 접근 편리

장점이 있다.

---

## 검토한 방향

### Common Util 구조

장점:
- 구현 속도 빠름
- 공통 로직 재사용 가능
- 접근 단순

단점:
- 책임 경계 모호
- 도메인 의미 약화
- 거대한 공통 패키지 발생

---

## 선택한 방향

공통 패키지보다
책임 기반 구조를 우선한다.

즉:
- 도메인 책임은 도메인 내부에 둔다.
- 역할이 명확한 객체를 사용한다.
- 실제 횡단 관심사만 global에 둔다.

---

## 선택 이유

### 책임 경계 약화

common/util 구조는 시간이 지나며:

    DateUtil
    OrderUtil
    ValidationHelper
    CommonManager

처럼 계속 비대해지기 쉽다.

그 결과:
- 책임 혼합
- 의존 증가
- 구조 탐색 어려움

문제가 발생할 수 있다.

---

### 도메인 의미 감소

프로젝트는:
- 도메인 중심 구조
- 책임 기반 설계

를 지향한다.

하지만 common/util 구조는:
- 기술 중심 분리
- 기능 중심 공통화

로 흐르기 쉽다.

---

### 재사용 목적 코드 증가

실제 사용보다:
- 미래 공통화
- 잠재 재사용

을 위한 코드가 증가할 수 있다.

그 결과:
- 과도한 추상화
- 의미 없는 유틸
- 숨겨진 의존성

문제가 발생할 수 있다.

---

## 허용 범위

아래 경우만 global 성격 패키지를 허용한다.

- exception
- config
- response
- logging
- security

즉:
- 횡단 관심사
- 인프라 성격

만 허용한다.

---

## 예시

좋은 예시:

    order/domain/OrderPolicy
    payment/application/PaymentValidator

지양하는 예시:

    common/ValidationUtil
    shared/CommonHelper

---

## 트레이드오프

### 중복 코드 증가 가능

초기에는 유사 코드가 반복될 수 있다.

---

### 공통 수정 비용 가능

동일 패턴 수정 시 여러 위치를 수정해야 할 수 있다.

---

## 현재 판단

현재 프로젝트는:

- 책임 명확성
- 도메인 응집도
- 구조 가독성

을 더 중요하게 판단한다.

따라서 Common Util 중심 구조는 채택하지 않는다.

---

## 재검토 신호

아래 상황이면 구조를 다시 검토한다.

- 같은 코드가 반복적으로 증가한다.
- 공통 정책 관리 비용이 커진다.
- 중복 수정 비용이 과도해진다.
- 도메인보다 공통 로직 유지 비용이 커진다.
