# Generic Response Wrapper

## 상태

rejected

---

## 문제 상황

모든 API 응답을
공통 Wrapper 객체로 감쌀지 고민했다.

예시:

    ApiResponse<T>
    CommonResponse<T>
    Result<T>

형태의 구조다.

---

## 검토한 방향

### Generic Response Wrapper 구조

예시:

    {
        "success": true,
        "data": ...
    }

장점:
- 응답 형식 통일 가능
- 공통 필드 추가 쉬움
- 프론트 처리 일관성 확보 가능

단점:
- HTTP 의미 약화
- 불필요한 중첩 증가
- 단순 응답까지 과도한 구조 강제

---

## 선택한 방향

HTTP 응답은 가능한 명시적으로 표현한다.

즉:
- 필요한 Response DTO를 직접 반환한다.
- HTTP 상태 코드를 우선 활용한다.
- 공통 Wrapper를 기본 전략으로 사용하지 않는다.

---

## 선택 이유

### HTTP 의미 유지

HTTP는 이미:
- 상태 코드
- 헤더
- 응답 구조

를 표현할 수 있다.

예시:

    200 OK
    201 Created
    404 Not Found

따라서:

    success: true

같은 필드를 모든 응답에 반복 추가하는 것은
의미 중복일 수 있다고 판단한다.

---

### 응답 가독성 유지

Wrapper 구조는:

    {
        "data": {
            ...
        }
    }

처럼 중첩을 강제한다.

단순 응답에서도:
- depth 증가
- JSON 구조 복잡화
- 실제 응답 의미 숨김

문제가 발생할 수 있다.

---

### 공통화 남용 방지

프로젝트는:
- 명시성
- 책임 기반 구조

를 더 중요하게 판단한다.

따라서:
- 모든 상황을 하나의 Wrapper로 일반화
보다
- 상황별 명확한 응답 표현

을 우선한다.

---

## 허용 범위

아래 상황에서는 공통 응답 구조를 일부 허용할 수 있다.

- ErrorResponse
- 페이지네이션 메타데이터
- 공통 API 규격 요구사항 존재

즉:
- 명확한 공통 의미
- 실제 반복 요구사항

이 있을 때만 제한적으로 사용한다.

---

## 예시

좋은 예시:

    OrderResponse
    UserResponse

지양하는 예시:

    ApiResponse<OrderResponse>

---

## 트레이드오프

### 응답 형식 통일성 감소 가능

엔드포인트마다 응답 구조가 달라질 수 있다.

---

### 프론트 처리 공통화 감소 가능

일부 공통 처리 로직 작성이 어려울 수 있다.

---

## 현재 판단

현재 프로젝트는:

- HTTP 의미 유지
- 응답 명시성
- 과도한 공통화 방지

를 더 중요하게 판단한다.

따라서 Generic Response Wrapper를 기본 전략으로 사용하지 않는다.

---

## 재검토 신호

아래 상황이면 구조를 다시 검토한다.

- API 응답 규격 통일 요구가 커진다.
- 공통 메타데이터가 반복된다.
- 응답 처리 비용이 증가한다.
- API Gateway 정책 요구사항이 생긴다.
